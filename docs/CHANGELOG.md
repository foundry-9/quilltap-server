#### Affordances

- **Personified-feature avatars — bundled source art for Aurora, Concierge, Host, Lantern, Librarian, Pascal, and Prospero, optimized to WebP**: New directory `public/images/avatars/` holds the canonical avatar asset for each named personified feature of the app (the ones that can "speak up" inside a chat — currently only the Lantern, with more planned). New CLAUDE.md section "Personified-feature avatars" documents the convention so future additions (Concierge announcements, Prospero affordances, etc.) know where the art lives and how to keep it sized correctly. (`public/images/avatars/`, `CLAUDE.md`)

#### Fixes

- **Background jobs — FAILED retries no longer strand forever when the processor auto-stops, and the Lantern routes around post-hoc image-moderation rejections through the Concierge**: Two linked bugs surfaced together on Friday (chat `af0642d1-c23a-4928-8184-4b33f4e2a6e7`, job `a9a56cbf-5661-446e-ad39-535f6cf1f212`). The user queued a story-background regeneration; Grok's image endpoint accepted the prompt but rejected the *generated image* post-hoc with `400 "Generated image rejected by content moderation."`. `markFailed` in `lib/database/repositories/background-jobs.repository.ts` correctly set `status='FAILED'` / `attempts=1/3` / `scheduledAt=2026-04-23T14:13:41Z` (60-second exponential-backoff retry), but two seconds later the processor's auto-stop path in `lib/background-jobs/processor.ts` fired — `claimNextJob()` returns only rows with `scheduledAt <= now`, which excluded the just-scheduled future retry, so the processor declared the queue empty, cleared the polling interval, and went quiet. Nothing else ever woke it back up: `ensureProcessorRunning` is only called from enqueue paths, so the retry sat idle indefinitely and the user's background regen never happened. Meanwhile the catch block at the image-generation call in `lib/background-jobs/handlers/story-background.ts` just re-threw the moderation error even though the Concierge already carried an `uncensoredImageProfileId` in `DangerousContentSettings` for exactly this case — the existing fallbacks for prompt crafting and appearance resolution checked it, but the generation call itself didn't. Three changes close both holes: (1) New `findNextScheduledAt()` on `BackgroundJobsRepository` — `$in: ['PENDING', 'FAILED']` + `$expr: { $lt: ['$attempts', '$maxAttempts'] }` sorted by `scheduledAt` ascending, limit 1 — returns the earliest future retry time across all retry-eligible jobs. (2) `processor.ts` gains a module-level `wakeUpTimer` with `armWakeUpTimer(scheduledAt)` (delay clamped to `[100ms, 5min]` so a far-future schedule doesn't pin a multi-hour timer and a past-due one gets a 100ms bounce) that `setTimeout`s a call to `ensureProcessorRunning()`. The auto-stop branch now queries `findNextScheduledAt()` before stopping and arms the timer if anything came back, so the FAILED job wakes the processor up exactly when its `scheduledAt` comes due. `startProcessor` and `stopProcessor` both call `clearWakeUpTimer()` so enqueues and shutdowns don't race a stale pending wake — explicit cleanup beats hoping nothing else starts the processor in the interim. (3) `story-background.ts` gains an `isImageModerationError(error)` helper matching the common phrasings (`"content moderation"`, `"content_policy"`, `"content policy"`, `"safety system"`, `"rejected by content"`, `"moderation_blocked"`) across Grok, OpenAI DALL-E, OpenRouter, and friends — keyword-based rather than status-code-based because every provider wraps its own error type and we don't want to care which SDK we caught. The catch block after `provider.generateImage` now checks moderation-shape + `dangerSettings.uncensoredImageProfileId` + "isn't the profile we just tried", and on match loads the uncensored profile, decrypts its API key through `connections.findApiKeyByIdAndUserId`, creates a second `createImageProvider` for its provider type, and retries with the same prompt/size/style. An `activeImageProfile` mutable tracks whichever profile ended up producing the final image, so the `generationModel` column on the resulting `files` row reflects reality (not the provider that rejected). The reroute logs both attempts through `logLLMCall` for Inspector visibility, and a reroute failure now throws `"Image generation failed after Concierge reroute: ..."` carrying the fallback's error message so the Inspector entry names the actual cause. Mirrors the existing prompt-crafting fallback at lines 430-468 and the appearance-resolution retry at lines 299-337 in the same file — same shape, same gating on presence of the uncensored profile rather than on `mode` (which already gates whether the Concierge is watching at all). Stranded jobs from before this fix will be picked up automatically on the next processor wake (which happens the moment any new job is enqueued, or on server restart). (`lib/database/repositories/background-jobs.repository.ts`, `lib/background-jobs/processor.ts`, `lib/background-jobs/handlers/story-background.ts`)

- **Physical backups — weekly, monthly, and yearly retention tiers now actually retain anything**: The tiered retention policy in `applyRetentionPolicy` (`lib/database/backends/sqlite/physical-backup.ts`) advertised "all for 7 days, weekly for 4 weeks, monthly for 12 months, yearly forever" and had code for every tier, but the weekly/monthly bucket loops iterated newest-first and `break`ed on the first match — so each bucket kept the *newest* backup in its age window. On the next day's retention pass, an even fresher backup entered the same window and became the new "kept" one; the previous day's pick was then missing from the `keep` set and got deleted at the bottom of the function. Net effect: nothing ever aged past ~8 days, so users who started the app daily ended up with ~7 daily + 1 "weekly" that was really just an 8-day-old daily. Weeks 2/3/4, month buckets, and the yearly tier never accumulated anything because no backup survived long enough to reach them. Phase 2 and Phase 3 now iterate oldest-first, keeping the *oldest* backup in each bucket — which has the most runway to age into the next tier on subsequent passes, instead of being aged out of its bucket and deleted the same day. A new regression test simulates 40 consecutive days of daily backup + retention runs with a stateful fs mock and asserts that all four weekly buckets plus the month-1 bucket end up populated; this would have failed loudly against the old code, which left only the 7-day daily window. (`lib/database/backends/sqlite/physical-backup.ts`, `__tests__/unit/lib/database/physical-backup.test.ts`)

- **Server — transient LLM-provider network hiccups no longer crash the whole process**: Observed on Friday (2026-04-22 16:12:45 UTC): a successful multi-character turn on GROK finished, post-turn fan-out enqueued a burst of INTER_CHARACTER_MEMORY / MEMORY_EXTRACTION / CONVERSATION_RENDER jobs, an outbound fetch() to a Cloudflare-fronted provider (IP `2606:4700::6812:373`) got its socket cut mid-stream with undici's `TypeError: terminated` (cause: `SocketError: other side closed`, `UND_ERR_SOCKET`, 7173 bytes written / 1694 read), and ~2.5 minutes of log silence followed before `combined.log` shows `[JobQueue] Processor started` with fresh config and `Initializing MCP connection manager` — i.e. the dev server actually died and restarted. The smoking gun was `setupSQLiteShutdownHandlers` in `lib/database/backends/sqlite/client.ts`: its `process.on('unhandledRejection', ...)` handler closes the SQLite connection, releases the instance lock, and calls `process.exit(1)` *unconditionally* for any rejection — originally wired that way so a genuinely bad state wouldn't corrupt SQLite at shutdown, but the blast radius is every transient socket close from every outbound fetch to every LLM provider, image API, or MCP server, since any of those can leak a `terminated` rejection past a generator's `for await` boundary if the body consumer doesn't wrap it in try/catch. Fix: the `unhandledRejection` handler now classifies rejections. A new `isRecoverableNetworkRejection` helper matches against a `RECOVERABLE_NETWORK_CODES` set covering undici stream errors (`UND_ERR_SOCKET`, `UND_ERR_CLOSED`, `UND_ERR_ABORTED`, `UND_ERR_BODY_TIMEOUT`, `UND_ERR_HEADERS_TIMEOUT`, `UND_ERR_CONNECT_TIMEOUT`, both content-length-mismatch variants) and POSIX socket errors (`ECONNRESET`, `ECONNREFUSED`, `ECONNABORTED`, `EPIPE`, `ETIMEDOUT`, `EHOSTUNREACH`, `ENETUNREACH`, `ENOTFOUND`, `EAI_AGAIN`), walks both `error.code` and `error.cause.code` since undici's `TypeError: terminated` carries the interesting code on its `cause` not itself, and falls back to matching `reason.name === 'TypeError' && reason.message === 'terminated'` for older undici versions whose cause chain is missing. Matching rejections log at `warn` level with the error message, code, and cause code, and the process keeps running. Non-matching rejections hit the original exit path unchanged — the "something is genuinely wrong" case (type errors in repository code, assertion failures, etc.) still takes the server down cleanly rather than silently corrupting state. `uncaughtException` is untouched because sync throws past this point usually mean V8-level corruption, not a Cloudflare edge closing a socket. (`lib/database/backends/sqlite/client.ts`)

- **Commonplace Book — housekeeping protection gate switches to a blended score**: The four-rule protection gate in `isProtectedMemory` — importance ≥ 0.7, `source === 'MANUAL'`, accessed within 3 months, or reinforced ≥ 5 times with a usefulness condition — has been replaced with a single blended protection score that combines four evidence streams: time-decayed LLM content importance (365-day half-life), a log-saturating reinforcement bonus, a graph-degree bonus from `relatedMemoryIds.length`, and a flat recent-access bonus for memories touched within the last 90 days. Memories scoring at or above **0.5** are protected; everything below is eligible for the cap-enforcement sweep. `source === 'MANUAL'` remains a hard override — explicit user intent is always durable. The motivation is a Friday-instance audit: on a character with 19,524 memories, **19,303 (98.9%)** were protected solely by `importance ≥ 0.7`, because the cheap-LLM extractor clusters almost every memory in the 0.7–0.9 band (0.8 was the modal value, with 8,826 memories). Under the old rule, the cap-enforcement sweep could only delete 1 memory on that character because almost everything hit the LLM-score bright line. Under the blended score, time decay and usage evidence do the discrimination the LLM number failed to — a 0.8-importance memory from 400 days ago with no reinforcement and no access drops below protection, while a reinforced well-linked recently-accessed memory stays even if the LLM rated it 0.3. #### Performance

- **Per-phase timing instrumentation on chat-turn context build**: After Phases 1 and 2 of the memory retrieval performance plan, the Friday instance (~20k memories on one character) still showed multi-minute main-thread pins between the compression-cache-hit log line and any subsequent context-build log line, with zero activity in between to tell us which phase was slow. Added `logger.debug`-level wall-clock timings at the phases we have reason to suspect: the parallel compression + proactive-recall `await Promise.all` in `lib/services/chat-message/orchestrator.service.ts`, the `buildMessageContext` call right after, and inside `buildContext` in `lib/chat/context-manager.ts` — `buildSystemPrompt`, the `countMessagesTokens` conversation pass, the per-character memory retrieval + format block (reports which path was taken: pre-searched, semantic-search, or skipped), and the inter-character memory retrieval block (multi-character chats only). `searchMemoriesSemantic` itself also emits a single `[Memory] Semantic search timings` line per call, breaking total wall time into embed, vector-search, and hydrate-plus-rank components. All logs are `debug` — invisible at the default `info` level, so there's no noise cost in normal operation, and a user chasing a stall can flip the log level at runtime to see where a chat turn is actually spending its time. The pattern matches the existing `[ContextManager]` and `[Memory]` log prefix convention so the entries sort together in log viewers. (`lib/memory/memory-service.ts`, `lib/services/chat-message/orchestrator.service.ts`, `lib/chat/context-manager.ts`)
- **Commonplace Book — vector search no longer sorts the entire index on every query**: On the Friday instance, the Phase 1 fix (`perf(memory): hydrate only matched memories in semantic search paths`) removed SQLCipher + Zod as the per-turn cost but exposed the next bottleneck: `CharacterVectorStore.search` in `lib/embedding/vector-store.ts` was pushing all 20k entries into a flat `VectorSearchResult[]`, then calling `Array.prototype.sort` with a user comparator, then slicing to `limit`. A `sample(1)` against a pinned `next-server` (100% CPU, silent log for 2+ minutes on a fresh chat turn) showed the entire main-thread stack under `Builtins_ArrayPrototypeSort` → `Builtins_ArrayTimSort` → `Builtins_MergeAt` → `Builtins_SortCompareUserFn` — the cost of a JS-comparator sort at 20k × called-per-agent-mode-sub-turn was pinning the event loop for minutes. `search` now routes to one of two paths: for small corpora (< 1000 entries or `limit * 4 >= entries.size`), the original linear scan + sort is retained — the heap's bookkeeping isn't worth it under that threshold; above it, a new hand-rolled bounded min-heap of size `limit` keeps only the top-K candidates during the scan, skipping the `VectorSearchResult` allocation entirely for any entry whose score can't beat the current worst. The heap is two parallel arrays (scores + results) so the hot `sift` comparison reads a primitive number, which matters because sift runs O(log k) per candidate across n entries. Output contract is unchanged: same top-K by score descending, same metadata, same dimension-mismatch fallback; tie-break order between entries with identical scores is not stable across paths (and was never contractually stable — V8's sort is stable but the heap's isn't, and neither matters for ranking semantics). Two parity tests added: a 5000-entry pseudorandom-unit-vector test asserting the heap and linear paths return identical score sequences, and a small-corpus test confirming the linear fallback still runs below the threshold. (`lib/embedding/vector-store.ts`, `__tests__/unit/lib/embedding/vector-store.test.ts`)
- **Commonplace Book — semantic memory retrieval no longer decrypts the whole corpus per chat turn**: Three hot paths in `lib/memory/memory-service.ts` — `searchMemoriesSemantic` (line 515), `findSimilarMemories` (line 711), and `findSimilarMemoriesWithEmbedding` (line 749) — were each calling `repos.memories.findByCharacterId(characterId)` to hydrate the top-K vector matches. That function decrypts and Zod-validates *every* memory row for the character in one synchronous pass, just to pluck ~60 hits out of a Map. On a character with ~20k memories, every chat turn paid 20k `MemorySchema.parse` calls and 20k SQLCipher row decrypts on the main thread — visible in the Salon as freezes during LLM turns and `Failed to fetch` on `/api/v1/session` and other concurrent requests. Each call site now does `findByIds(vectorResults.map(r => r.id))` instead, mirroring the pattern the Memory Gate write path (`lib/memory/memory-gate.ts:161`) has used since 4.3. Per-turn retrieval cost now scales with *hit count* rather than corpus size: on a 20k-memory character, the worst of the three reads drops from 20k decrypted-and-validated rows to ~60. No change to returned memory set, ranking, filter semantics, or dimension-mismatch fallback behavior. `generateMissingEmbeddings` (line 781) and `rebuildVectorIndex` (line 852) still do full scans — they legitimately need the whole corpus (the former filters for null embeddings; the latter rebuilds the whole index) and are cold paths not on the chat turn hot path. (`lib/memory/memory-service.ts`)
- **Commonplace Book — no-embedding-no-write gate, raised thresholds, and a near-duplicate skip tier**: Three interlocking changes to the Memory Gate, driven by the same Friday-instance audit: of 32,364 memories, 13,174 (41%) had `NULL` embeddings and were therefore invisible to every future gate check, letting phrase-variant duplicates like `"Sunny wears teal glasses."` accumulate ten literal copies. (1) When the gate can't generate an embedding for a candidate, it now retries once after a 500 ms delay and, on second failure, returns a new `SKIP_EMBEDDING_FAILED` decision instead of falling through to the old keyword-overlap gate. No row is written, because a row without an embedding is a permanent dedup hole. The `keywordBasedGate` fallback function has been deleted outright — it was the sole producer of those NULL-embedding rows and its lone caller has been replaced. (2) A new `NEAR_DUPLICATE_THRESHOLD = 0.90` tier sits above the existing merge band: candidates whose best vector-search neighbor scores at or above 0.90 now resolve to a new `SKIP_NEAR_DUPLICATE` decision that neither inserts nor reinforces — the observation is absorbed silently into the existing memory. (3) `MERGE_THRESHOLD` rises from 0.80 to 0.85 so reinforcement is reserved for genuinely duplicate-enough matches rather than merely very-similar ones; the 0.70–0.85 band now produces `INSERT_RELATED` (was 0.70–0.80). `MemoryGateOutcome.memory` is now `Memory | null` to carry the SKIP states, and `createMemoryWithEmbedding` (the manual POST path) returns `null` on `SKIP_EMBEDDING_FAILED` so the API surfaces a clear 500 rather than silently no-op'ing. The three switch blocks in `processMessageForMemory` and `processInterCharacterMemory` handle both SKIP actions with distinct debug-log markers so the per-message `debugMemoryLogs` can distinguish "skipped for similarity" from "skipped because embedding failed." (`lib/memory/memory-gate.ts`, `lib/memory/memory-service.ts`, `lib/memory/memory-processor.ts`, `lib/memory/index.ts`, `app/api/v1/memories/route.ts`, `__tests__/unit/lib/memory/memory-gate.test.ts`, `__tests__/unit/lib/memory/memory-processor-regression.test.ts`)
- **Commonplace Book — hard candidate cap and targeted gate lookup**: Two compounding fixes along the memory-write path, motivated by a Friday-instance audit that found 32k memories on one instance with 7,717 added in a single day and individual messages producing up to 88 memory rows. (1) `parseMemoryCandidateArray` now slices significant candidates to `HARD_CANDIDATE_CAP = 5` after the significance filter, and each call site derives `maxMemories` through `resolveMaxMemories()` which clamps the prior `Math.ceil(resolvedMaxTokens / 8000)` formula to the same ceiling — so a 128K-output cheap-LLM profile is told in the prompt to return at most 5 and *also* can't slip more past the parser if it ignores the instruction. Defense-in-depth across the three extraction paths (user memories, character-self memories, inter-character memories). (2) The Memory Gate's pre-write similarity check previously called `repos.memories.findByCharacterId(characterId)` to resolve the top-5 vector-search hits into full `Memory` rows — an O(N) scan on every single insert that loads 20K rows just to `.get()` five of them. A new repository method `findByIds(ids: string[])` runs a targeted `WHERE id IN (...)` filter through the existing `findByFilter` abstraction, and the gate now passes only `results.map(r => r.id)` into it. Net effect on a character with 20K memories: the per-insert read drops from a full-table scan to a five-row primary-key lookup, with identical dedup semantics. - **Memory extraction — halve the per-call candidate cap**: Each chat turn now extracts at most `ceil(maxTokens / 8000)` memory candidates per extraction call, down from `ceil(maxTokens / 4000)`. For a 32K-context cheap-LLM profile that drops the per-call ceiling from 8 to 4; for an 8K profile (the default fallback) the ceiling drops from 2 to 1. Extraction still runs three calls per turn (user-memories, character-self-memories, inter-character-memories), but a multi-character chat that previously fanned out 30–40 memory writes per turn — each of which serialises an Ollama embedding ~1 s — should now do roughly half that work without losing the salient items, since the LLM was already prioritising significance and just had headroom to over-collect. (`lib/memory/cheap-llm-tasks/memory-tasks.ts`)
- **Vector search — Float32Array, unit-vector cosine, and a mount-chunk cache**: Three compounding fixes along the embedding hot path, motivated by a Friday-instance chat turn that stalled on the order of a minute after "Loading tools for Friday…". (1) `blobToEmbedding` no longer boxes each Float32 BLOB into a `number[]` of doubles on read — it returns a proper `Float32Array`, and that type now flows end-to-end through `EmbeddingResult`, every embedding-carrying Zod schema (`memories`, `conversation_chunks`, `doc_mount_chunks`, `help_docs`, `vector_entries`), every `updateEmbedding`/`addEntries` repository signature, and every `cosineSimilarity` call site. Same BLOB on disk, four-times-less RAM and sharply tighter JIT loops in memory. (2) All stored embeddings are now L2-normalised at write time inside the single choke-point `generateEmbeddingForUser`, so `cosineSimilarity(a, b)` reduces to a single-pass dot product — no per-comparison `Math.sqrt` calls. The new migration `normalize-embeddings-unit-vectors-v1` walks every embedding in the main DB and the mount-index DB, normalizing any non-unit vector in place (idempotent — re-running skips vectors already within 1e-4 of unit length). (3) A new `lib/mount-index/mount-chunk-cache.ts` keeps embedded document-mount chunks resident in process memory keyed by mount point, mirroring the shape of `VectorStoreManager`. Loading is lazy on first search; invalidation fires when a chunk is re-embedded (`handleMountChunkEmbedding`), chunks are deleted (`deleteByFileId` / `deleteByMountPointId`), or an embedding reindex is enqueued. `searchDocumentChunks` consumes the cache instead of re-decoding thousands of BLOBs on every turn. (4) `GET /api/v1/mount-points` no longer hydrates every embedded chunk across every store just to populate `embeddedChunkCount` in the Scriptorium settings UI — a new `countEmbeddedByMountPointIds` runs a single `GROUP BY` query instead. Net effect on the observed slowdown: the per-turn mount-chunk decode drops from ~1 s + 33 M multiplies-plus-sqrts to a cache hit + one dot-product pass.
