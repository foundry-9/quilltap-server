# Quilltap Changelog

## Recent Changes

### 4.3-dev

#### Fixes

- **Commonplace Book — *Run housekeeping now* no longer fails validation**: The button on the Memory Housekeeping card posted to `POST /api/v1/memories?action=housekeep`, but that endpoint's Zod schema requires `characterId` and the card sent an empty body — so every click bounced as a validation error before a single log line could fire. The synchronous per-character handler was the wrong target for a button whose copy (and the help file) promises a background sweep across every character. Added a new `POST /api/v1/memories?action=housekeep-sweep` that wraps `enqueueMemoryHousekeeping(user.id, { reason: 'manual' })` — the existing queue helper already accepts an empty payload as "sweep every character owned by userId" and dedupes against in-flight jobs for the same `(userId, characterId)` pair. The card now hits the new action, so the toast's "Housekeeping job enqueued — it will run in the background" claim is finally true. (`app/api/v1/memories/route.ts`, `components/tools/memory-housekeeping-card.tsx`)

#### Improvements

- **Memory recall — tuned the retrieval knobs toward reinforced, higher-importance memories**: Four defaults in the per-turn Commonplace Book lookup that feeds the system prompt have been rebalanced. `maxMemories` bumps from 10 to 18 so each `## Relevant Memories` block has more room (still bounded by the memory token budget that trims the tail). `minMemoryImportance` rises from 0.3 to 0.5, raising the floor on what even gets into the candidate pool. The vector-store overshoot widens from `limit × 2` to `limit × 3`, giving the post-fetch filters (importance, source, scoring) more headroom before they starve the final slice. And the hybrid ranking used by both the semantic path and the text-search fallback flips from `0.6 · cosineSimilarity + 0.4 · effectiveWeight` to `0.4 · cosineSimilarity + 0.6 · effectiveWeight` — so reinforcement count and time-decay floor now drive ordering, with raw query similarity as the secondary factor. Combined effect: the retrieval pool is narrower at the bottom, wider at the top, and ranked by what the character has had repeatedly confirmed rather than what happens to phrase-match the current user turn. (`lib/chat/context-manager.ts`, `lib/services/chat-message/context-builder.service.ts`, `lib/memory/memory-service.ts`)

#### Features

- **Commonplace Book — inter-character memory handler routes through the Memory Gate**: The INTER_CHARACTER_MEMORY job handler previously called the cheap-LLM extraction directly and then wrote every candidate to `repos.memories.create` inline, bypassing `runMemoryGate` entirely. That meant none of the 4.3 gate improvements — NEAR_DUPLICATE skip at 0.90, MERGE at 0.85, SKIP_EMBEDDING_FAILED, the retry-once-then-drop embedding policy, reinforcement on match, related-memory linking, or the post-write auto-housekeeping watermark — applied to observer→subject memories in multi-character chats. Those memories landed as duplicates with NULL embeddings in exactly the way the single-character path used to, only still. The handler has been reduced to a 70-line adapter that loads the connection profile, chat settings, danger resolution, and chat context, then hands off to `processInterCharacterMemory` in `lib/memory/memory-processor`. That function already contained the full gate-aware code path for inter-character extraction — introduced alongside the single-character processor but never wired to production — including the rate-limit check from the 4.3 extraction-limits work. Net effect: one code path now covers all three extraction flavors (user memories, character-self memories, inter-character memories), with identical gate behavior. New unit test suite at `__tests__/unit/background-jobs/inter-character-memory-handler.test.ts` covers five cases (delegation, missing profile, missing chat settings, dangerous-chat propagation, graceful failure). Debug-log persistence on the source message is deliberately left out of the inter-character path for now — the single-character handler's pattern overwrites `debugMemoryLogs` on each write, which would race when multiple observers all log against the same source message; the prior handler didn't persist them either, so this is behavior-preserving. (`lib/background-jobs/handlers/inter-character-memory.ts`, `__tests__/unit/background-jobs/inter-character-memory-handler.test.ts`)

A new lightweight endpoint `GET /api/v1/memories?action=character-memory-counts` powers the list — rather than loading the full enriched character payload from `/api/v1/characters` and paying for the per-character chat lookup that enrichment does, this returns only `{id, name, memoryCount}` triples so the Housekeeping card stays fast even on instances with many characters. Saves fire on blur via the existing `?action=housekeeping-config` POST path; the list sorts characters by memory count descending so the ones most likely to need attention appear first. - **Commonplace Book — *Repair Missing Embeddings* admin card**: A new card in the Commonplace Book tab (second from the top, between *Embedding Profiles* and *Memory Housekeeping*) lists how many memories across every character are still missing an embedding, how many embedding jobs are currently in flight for this user, and offers a button to enqueue up to 500 more at a time. - **Commonplace Book — per-hour extraction rate limit with a graduated importance floor**: A companion guard to auto-housekeeping that applies back-pressure at the front of the memory pipeline rather than the back. When enabled, the limiter counts how many memories a character has accrued in the trailing hour and — as volume approaches the cap — tightens what the extraction pass is allowed to commit: below 70% of the cap, candidates flow through unchanged; between 70% and 100%, candidates whose importance falls below the soft floor are dropped; at or above 100%, extraction is skipped entirely for this exchange and logged to the per-message `debugMemoryLogs` so the user can see why their memories stopped appearing. The limiter applies to all three extraction paths — user memories, character memories (both in `processMessageForMemory`), and inter-character memories (in the background-jobs `inter-character-memory.ts` handler, which writes directly rather than going through the unused `processInterCharacterMemory` entry point). **Shipped disabled by default** and not yet exposed in the UI; it is the API-only backstop for instances whose gate improvements alone are not enough. Complementary to auto-housekeeping: housekeeping is retrospective and the rate limit is prospective. #### Performance

- **Commonplace Book — no-embedding-no-write gate, raised thresholds, and a near-duplicate skip tier**: Three interlocking changes to the Memory Gate, driven by the same Friday-instance audit: of 32,364 memories, 13,174 (41%) had `NULL` embeddings and were therefore invisible to every future gate check, letting phrase-variant duplicates like `"Sunny wears teal glasses."` accumulate ten literal copies. (1) When the gate can't generate an embedding for a candidate, it now retries once after a 500 ms delay and, on second failure, returns a new `SKIP_EMBEDDING_FAILED` decision instead of falling through to the old keyword-overlap gate. No row is written, because a row without an embedding is a permanent dedup hole. The `keywordBasedGate` fallback function has been deleted outright — it was the sole producer of those NULL-embedding rows and its lone caller has been replaced. (2) A new `NEAR_DUPLICATE_THRESHOLD = 0.90` tier sits above the existing merge band: candidates whose best vector-search neighbor scores at or above 0.90 now resolve to a new `SKIP_NEAR_DUPLICATE` decision that neither inserts nor reinforces — the observation is absorbed silently into the existing memory. (3) `MERGE_THRESHOLD` rises from 0.80 to 0.85 so reinforcement is reserved for genuinely duplicate-enough matches rather than merely very-similar ones; the 0.70–0.85 band now produces `INSERT_RELATED` (was 0.70–0.80). `MemoryGateOutcome.memory` is now `Memory | null` to carry the SKIP states, and `createMemoryWithEmbedding` (the manual POST path) returns `null` on `SKIP_EMBEDDING_FAILED` so the API surfaces a clear 500 rather than silently no-op'ing. The three switch blocks in `processMessageForMemory` and `processInterCharacterMemory` handle both SKIP actions with distinct debug-log markers so the per-message `debugMemoryLogs` can distinguish "skipped for similarity" from "skipped because embedding failed." (`lib/memory/memory-gate.ts`, `lib/memory/memory-service.ts`, `lib/memory/memory-processor.ts`, `lib/memory/index.ts`, `app/api/v1/memories/route.ts`, `__tests__/unit/lib/memory/memory-gate.test.ts`, `__tests__/unit/lib/memory/memory-processor-regression.test.ts`)
- **Commonplace Book — hard candidate cap and targeted gate lookup**: Two compounding fixes along the memory-write path, motivated by a Friday-instance audit that found 32k memories on one instance with 7,717 added in a single day and individual messages producing up to 88 memory rows. (1) `parseMemoryCandidateArray` now slices significant candidates to `HARD_CANDIDATE_CAP = 5` after the significance filter, and each call site derives `maxMemories` through `resolveMaxMemories()` which clamps the prior `Math.ceil(resolvedMaxTokens / 8000)` formula to the same ceiling — so a 128K-output cheap-LLM profile is told in the prompt to return at most 5 and *also* can't slip more past the parser if it ignores the instruction. Defense-in-depth across the three extraction paths (user memories, character-self memories, inter-character memories). (2) The Memory Gate's pre-write similarity check previously called `repos.memories.findByCharacterId(characterId)` to resolve the top-5 vector-search hits into full `Memory` rows — an O(N) scan on every single insert that loads 20K rows just to `.get()` five of them. A new repository method `findByIds(ids: string[])` runs a targeted `WHERE id IN (...)` filter through the existing `findByFilter` abstraction, and the gate now passes only `results.map(r => r.id)` into it. Net effect on a character with 20K memories: the per-insert read drops from a full-table scan to a five-row primary-key lookup, with identical dedup semantics. (`lib/memory/cheap-llm-tasks/memory-tasks.ts`, `lib/memory/memory-gate.ts`, `lib/database/repositories/memories.repository.ts`, `__tests__/unit/lib/memory/cheap-llm-tasks.test.ts`)
- **Memory extraction — async via the existing background-job queue**: Memory extraction (single-character, inter-character, and user-controlled-character) was previously inlined in `finalizeMessageResponse`, holding the request open for 30–40 s of serial Ollama embedding work after the SSE `done` event had already been sent — which made the next chat turn feel like it was still chewing on the previous one. The chat path now enqueues `MEMORY_EXTRACTION` and `INTER_CHARACTER_MEMORY` jobs (one per observer→subject pair) instead of awaiting `processMessageForMemoryAsync` directly. The job types, handlers, and the **Mem** badge in `<QueueStatusBadges />` already existed and were just unused for this path; the work now drains in the background and surfaces in the header counter so it's visible. The handler grew the missing `userCharacterName`, `characterPronouns`, `allCharacterNames`, and `allCharacterPronouns` payload fields so the worker has the same context the inline trigger had, plus a token-tracking `createMemoryExtractionEvent` emit that previously lived in the trigger callback. (`lib/services/chat-message/memory-trigger.service.ts`, `lib/background-jobs/queue-service.ts`, `lib/background-jobs/handlers/memory-extraction.ts`, `lib/background-jobs/handlers/inter-character-memory.ts`)
- **Memory extraction — halve the per-call candidate cap**: Each chat turn now extracts at most `ceil(maxTokens / 8000)` memory candidates per extraction call, down from `ceil(maxTokens / 4000)`. For a 32K-context cheap-LLM profile that drops the per-call ceiling from 8 to 4; for an 8K profile (the default fallback) the ceiling drops from 2 to 1. Extraction still runs three calls per turn (user-memories, character-self-memories, inter-character-memories), but a multi-character chat that previously fanned out 30–40 memory writes per turn — each of which serialises an Ollama embedding ~1 s — should now do roughly half that work without losing the salient items, since the LLM was already prioritising significance and just had headroom to over-collect. (`lib/memory/cheap-llm-tasks/memory-tasks.ts`)
- **Vector search — Float32Array, unit-vector cosine, and a mount-chunk cache**: Three compounding fixes along the embedding hot path, motivated by a Friday-instance chat turn that stalled on the order of a minute after "Loading tools for Friday…". (1) `blobToEmbedding` no longer boxes each Float32 BLOB into a `number[]` of doubles on read — it returns a proper `Float32Array`, and that type now flows end-to-end through `EmbeddingResult`, every embedding-carrying Zod schema (`memories`, `conversation_chunks`, `doc_mount_chunks`, `help_docs`, `vector_entries`), every `updateEmbedding`/`addEntries` repository signature, and every `cosineSimilarity` call site. Same BLOB on disk, four-times-less RAM and sharply tighter JIT loops in memory. (2) All stored embeddings are now L2-normalised at write time inside the single choke-point `generateEmbeddingForUser`, so `cosineSimilarity(a, b)` reduces to a single-pass dot product — no per-comparison `Math.sqrt` calls. The new migration `normalize-embeddings-unit-vectors-v1` walks every embedding in the main DB and the mount-index DB, normalizing any non-unit vector in place (idempotent — re-running skips vectors already within 1e-4 of unit length). (3) A new `lib/mount-index/mount-chunk-cache.ts` keeps embedded document-mount chunks resident in process memory keyed by mount point, mirroring the shape of `VectorStoreManager`. Loading is lazy on first search; invalidation fires when a chunk is re-embedded (`handleMountChunkEmbedding`), chunks are deleted (`deleteByFileId` / `deleteByMountPointId`), or an embedding reindex is enqueued. `searchDocumentChunks` consumes the cache instead of re-decoding thousands of BLOBs on every turn. (4) `GET /api/v1/mount-points` no longer hydrates every embedded chunk across every store just to populate `embeddedChunkCount` in the Scriptorium settings UI — a new `countEmbeddedByMountPointIds` runs a single `GROUP BY` query instead. Net effect on the observed slowdown: the per-turn mount-chunk decode drops from ~1 s + 33 M multiplies-plus-sqrts to a cache hit + one dot-product pass. - **Orphaned PROCESSING jobs blocking new work**: On startup, all PROCESSING jobs are now killed (DEAD) instead of reset to PENDING, since no job can legitimately be mid-flight at server start. Prevents stale retries from blocking fresh batch-inserted jobs. Embedding concurrency also guards against over-claiming via early bail when slots are full, with proactive back-fill when a slot frees.
- **Embedding BLOB Registration**: Fixed race condition where memory embeddings could be stored as JSON text instead of Float32 BLOBs, causing dimension mismatches during vector search. BLOB columns for `memories`, `vector_entries`, and `conversation_chunks` are now registered at database initialization time rather than lazily in individual repositories.
