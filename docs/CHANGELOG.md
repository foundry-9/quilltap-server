# Quilltap Changelog

## Recent Changes

### 3.3-dev

- feat: **Copy image to clipboard** — add a "Copy to clipboard" button to all three full-screen image viewer modals (ImageModal, ChatGalleryImageViewModal, ImageDetailModal); uses `navigator.clipboard.write` with `ClipboardItem` to copy the actual image data; button appears between Download and Close in the top-right action bar with a two-overlapping-squares icon
- fix: **Image viewer buttons invisible in dark mode** — replace `bg-primary/10` and `text-primary-foreground` on all image overlay action buttons with `qt-bg-overlay-btn` and `qt-text-overlay` semantic classes designed for dark overlays; affects tag, download, copy, close, navigation, and delete buttons across all three image viewer modals
- refactor: **Registry singleton class hierarchy** — extract shared code from the 7 singleton registry classes (ProviderRegistry, ModerationProviderRegistry, SearchProviderRegistry, ToolRegistry, PluginRegistry, ThemeRegistry, RoleplayTemplateRegistry) into a three-level abstract class hierarchy: `AbstractRegistry` (global state persistence, `isInitialized()`, `reset()`), `AbstractMapRegistry` (Map-based item/error storage), and `AbstractProviderRegistry` (shared provider registration, lookup, initialization, validation, hot-loading, stats, errors, and state export); the three provider registries were ~90% identical copy-paste code and are now reduced to only their unique methods; all public API signatures and exports preserved with zero breaking changes
- fix: **Character Optimizer crashes when LLM omits frequency field** — the `FrequencyBadge` component in the analysis summary called `.toLowerCase()` on the `frequency` property of behavioral patterns without a null guard; when the LLM returns a pattern object without a `frequency` field, this throws a TypeError; add nullish coalescing fallback to empty string
- fix: **Character Optimizer crashes on truncated LLM JSON** — when the LLM hit the `maxTokens` limit during character optimization, the response was truncated mid-JSON, causing `parseLLMJson()` to throw "Unexpected end of JSON input"; add `repairTruncatedJson()` to `parseLLMJson` that closes unclosed strings, brackets, and braces, and strips dangling keys/commas; double `maxTokens` limits for optimizer calls (analysis 4000→8000, suggestions 8000→16000); add detailed error logging around both JSON parse calls for future debugging
- fix: **Update-participant action rejects valid requests** — the `update-participant` action handler parsed the raw request body directly against the validation schema, but the frontend wraps the payload in an `{ updateParticipant: { ... } }` envelope; the handler now unwraps the envelope when present, fixing 400 errors when changing connection profiles, status, or other participant settings from the sidebar
- fix: **Footer shows wrong data directory when QUILLTAP_DATA_DIR is overridden** — `getHostDataDir()` checked `QUILLTAP_HOST_DATA_DIR` unconditionally, but that env var may be stale in local environments (e.g., set by Electron for a different instance); now only consults `QUILLTAP_HOST_DATA_DIR` inside Lima VM or Docker containers where the internal/host path split is meaningful; in local mode, always uses `getBaseDataDir()` which respects the current `QUILLTAP_DATA_DIR`
- docs: **Database DDL reference** — add `docs/developer/DDL.md` with complete schema documentation for both SQLite databases (main and LLM logs): CLI access instructions, encryption details, runtime PRAGMAs, and full CREATE TABLE/INDEX statements for all 31 tables; add DDL update requirement to commit checklist and CLAUDE.md data change guidelines; register in documentation index
- chore: Remove unused eslint-disable directives in context compression unit tests
- feat: **Four-state character participation system** — characters in multi-character chats now have four participation states instead of a binary active/inactive toggle: **active** (speaks and roleplays normally), **silent** (gets turns but must not speak out loud — can have inner thoughts, physical actions, and reactions but no audible dialogue), **absent** (turn manager skips them while they're away), and **removed** (no longer part of the chat); status changes are recorded as `STATUS_CHANGE` system events and all LLM-controlled characters are notified in their next turn's prompt when another character changes status; prompts include a participant status guide explaining what each mode means; the responding character receives explicit silent-mode instructions when in silent status; messages from silent characters are styled distinctly in the UI with dotted borders and muted teal/sage tones (similar to whisper styling but visually distinct); the old eye icon toggle is replaced with a status dropdown on participant cards; silent styling is persisted per-message via `isSilentMessage` flag (new `chat_messages` column) so changing a character's status does not retroactively restyle old messages; database migrations convert existing `isActive` boolean to the new `status` enum and add the `isSilentMessage` column; scene state tracking now filters out absent and removed characters so they don't appear in the LLM's scene description; participant cards show Silent/Absent badges after the AI badge, and avatar overlays indicate status when the sidebar is collapsed; `isActive` remains as a computed backward-compatible field
- docs: **Help file updates for four-state participation** — update chat-participants.md (status dropdown replaces eye icon toggle, mini avatar status overlays, Silent/Absent badges), chat-multi-character.md (new "Character Participation States" section with all four states and change notification behavior), chat-turn-manager.md (eligibility rules, turn indicators split into Silent/Absent categories), and chat-settings.md (compression section documents per-participant caching and system prompt preservation)
- chore: **Theme storybook v1.0.26** — add silent message styling demos (`.qt-chat-message-silent`), participant status badge stories, and status indicator examples; all 5 bundled themes updated with `--qt-chat-silent-*` CSS variables in complementary muted teal/sage/green tones for both light and dark modes
- fix: **Per-character compression cache and removal of system prompt compression** — context compression is now per-participant in multi-character chats, keyed by `chatId:participantId` instead of just `chatId`; each character gets their own compressed history reflecting their actual message visibility (filtered by join time, whispers, and absence); system prompt compression has been removed entirely — only conversation history is compressed, because each character needs their own uncompressed identity/personality prompt; the "System Prompt Compression Target" slider has been removed from the Foundry settings; backward-compatible database format migration handles old single-cache format transparently
- fix: **Multi-character chat identity confusion from compression cache** — in multi-character chats with context compression enabled, the compressed system prompt from the first character to respond was cached per-chat and reused for all subsequent characters in the chain, causing every character to receive the first character's identity and personality in their prompt; the compression cache stores a `compressedSystemPrompt` keyed by `chatId` (not per-character), so the second, third, etc. characters in a turn cycle received the wrong system prompt; fix by skipping the cached `compressedSystemPrompt` in multi-character chats and always using the freshly-built system prompt for the current responding character; compressed conversation history (which is character-agnostic) continues to be shared from the cache
- fix: **Deactivated characters can be re-added via Add Character dialog** — characters that were deactivated (eye icon toggle) in a multi-character chat still appeared in the "Add new character" dialog because the exclusion filter only checked `isActive`, which is `false` for both deactivated and fully removed characters; fix by filtering on `removedAt` instead — characters without a `removedAt` timestamp are still participants (active or silenced) and excluded from the dialog, while truly removed characters remain re-addable; add matching API-side guard in the add-participant action to reject adding a character that is deactivated but not removed; add `removedAt` to the frontend `Participant` type
- fix: **Greeting generation falls back to Concierge uncensored provider on content filter** — when a provider's content filter strips the auto-generated first message (tokens consumed but empty response), the greeting generator now detects the pattern and falls back to the Concierge system's configured uncensored provider (when AUTO_ROUTE mode is enabled); previously, two filtered attempts would silently fall through to a generic static greeting; `generateGreetingMessage` now returns a `GreetingResult` with a `contentFilterDetected` flag; retry sequence is: full context → without memories → Concierge uncensored provider → final plain retry with delay
- fix: **Nudge causes duplicate character responses in multi-character chat** — clicking the Nudge button on a character's participant card would produce two near-identical responses from that character; `handleNudge` was both adding the participant to the server-side turn queue via `callTurnAction('nudge')` AND triggering an immediate response via `triggerContinueMode`; the chain loop in `shouldChainNext` would then pop the queued entry and generate a second response; fix by removing the redundant queue add from `handleNudge` (the direct `triggerContinueMode` call is sufficient), and adding a server-side safety net in `shouldChainNext` that skips any queue entry matching the participant who just spoke
- feat: **Memory recap at chat start** — characters now receive a narrative summary of their recent memories when a chat begins or when they join an existing chat; the system fetches the last 50 high-importance, 20 medium-importance, and 10 low-importance memories, sends them to the cheap LLM for first-person narrative summarization, and injects the result as a "What You Remember" section in the system prompt after character personality notes but before the identity reinforcement lockdown; triggers automatically on the first message of any chat and when a character responds for the first time in a multi-character chat; supports uncensored provider fallback for dangerous chats where the cheap LLM may refuse to process sensitive memories
- fix: **Multi-character chat participant removal loses message attribution** — removing a character from a multi-character chat hard-deleted the participant from the array, causing all their messages to lose identity and fall back to the first remaining character or user; `removeParticipant()` now soft-deletes by setting `isActive: false` and a new `removedAt` timestamp, preserving the participant record so `getMessageAvatar()` and `getParticipantById()` continue to resolve the correct name and avatar for historical messages; adds `removedAt` field to `ChatParticipantSchema`, `ChatParticipantBaseSchema`, and all enrichment interfaces
- fix: **Multi-character chat runaway chaining (all-LLM misdetection)** — chats without an explicit `controlledBy: 'user'` participant (which includes all chats created via the current API) were misidentified as "all-LLM", causing `selectNextSpeaker()` to endlessly cycle characters instead of stopping for user input; the orchestrator now passes a sentinel user participant ID when the user sends a message, `shouldChainNext()` checks for USER messages in history before applying all-LLM pause logic, and the frontend `isAllLLM` memo also checks message history; truly all-LLM chats (no user messages) continue to work as before
- fix: **Re-adding a previously removed character creates duplicate participant** — when adding a character that was previously removed (soft-deleted) from a chat, the add-participant handler now reactivates the existing participant entry instead of creating a new one; prevents duplicate participant records that could confuse turn management and message attribution
