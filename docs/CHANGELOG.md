# Quilltap Changelog

## Recent Changes

### 4.3-dev

#### Performance

- **Memory extraction — async via the existing background-job queue**: Memory extraction (single-character, inter-character, and user-controlled-character) was previously inlined in `finalizeMessageResponse`, holding the request open for 30–40 s of serial Ollama embedding work after the SSE `done` event had already been sent — which made the next chat turn feel like it was still chewing on the previous one. The chat path now enqueues `MEMORY_EXTRACTION` and `INTER_CHARACTER_MEMORY` jobs (one per observer→subject pair) instead of awaiting `processMessageForMemoryAsync` directly. The job types, handlers, and the **Mem** badge in `<QueueStatusBadges />` already existed and were just unused for this path; the work now drains in the background and surfaces in the header counter so it's visible. The handler grew the missing `userCharacterName`, `characterPronouns`, `allCharacterNames`, and `allCharacterPronouns` payload fields so the worker has the same context the inline trigger had, plus a token-tracking `createMemoryExtractionEvent` emit that previously lived in the trigger callback. (`lib/services/chat-message/memory-trigger.service.ts`, `lib/background-jobs/queue-service.ts`, `lib/background-jobs/handlers/memory-extraction.ts`, `lib/background-jobs/handlers/inter-character-memory.ts`)
- **Memory extraction — halve the per-call candidate cap**: Each chat turn now extracts at most `ceil(maxTokens / 8000)` memory candidates per extraction call, down from `ceil(maxTokens / 4000)`. For a 32K-context cheap-LLM profile that drops the per-call ceiling from 8 to 4; for an 8K profile (the default fallback) the ceiling drops from 2 to 1. Extraction still runs three calls per turn (user-memories, character-self-memories, inter-character-memories), but a multi-character chat that previously fanned out 30–40 memory writes per turn — each of which serialises an Ollama embedding ~1 s — should now do roughly half that work without losing the salient items, since the LLM was already prioritising significance and just had headroom to over-collect. (`lib/memory/cheap-llm-tasks/memory-tasks.ts`)
- **Vector search — Float32Array, unit-vector cosine, and a mount-chunk cache**: Three compounding fixes along the embedding hot path, motivated by a Friday-instance chat turn that stalled on the order of a minute after "Loading tools for Friday…". (1) `blobToEmbedding` no longer boxes each Float32 BLOB into a `number[]` of doubles on read — it returns a proper `Float32Array`, and that type now flows end-to-end through `EmbeddingResult`, every embedding-carrying Zod schema (`memories`, `conversation_chunks`, `doc_mount_chunks`, `help_docs`, `vector_entries`), every `updateEmbedding`/`addEntries` repository signature, and every `cosineSimilarity` call site. Same BLOB on disk, four-times-less RAM and sharply tighter JIT loops in memory. (2) All stored embeddings are now L2-normalised at write time inside the single choke-point `generateEmbeddingForUser`, so `cosineSimilarity(a, b)` reduces to a single-pass dot product — no per-comparison `Math.sqrt` calls. The new migration `normalize-embeddings-unit-vectors-v1` walks every embedding in the main DB and the mount-index DB, normalizing any non-unit vector in place (idempotent — re-running skips vectors already within 1e-4 of unit length). (3) A new `lib/mount-index/mount-chunk-cache.ts` keeps embedded document-mount chunks resident in process memory keyed by mount point, mirroring the shape of `VectorStoreManager`. Loading is lazy on first search; invalidation fires when a chunk is re-embedded (`handleMountChunkEmbedding`), chunks are deleted (`deleteByFileId` / `deleteByMountPointId`), or an embedding reindex is enqueued. `searchDocumentChunks` consumes the cache instead of re-decoding thousands of BLOBs on every turn. (4) `GET /api/v1/mount-points` no longer hydrates every embedded chunk across every store just to populate `embeddedChunkCount` in the Scriptorium settings UI — a new `countEmbeddedByMountPointIds` runs a single `GROUP BY` query instead. Net effect on the observed slowdown: the per-turn mount-chunk decode drops from ~1 s + 33 M multiplies-plus-sqrts to a cache hit + one dot-product pass. (`lib/embedding/embedding-service.ts`, `lib/database/backends/sqlite/json-columns.ts`, `lib/database/backends/sqlite/query-translator.ts`, `lib/schemas/{memory,scriptorium,mount-index,vector-indices,help-doc}.types.ts`, `lib/help-search.types.ts`, `lib/embedding/vector-store.ts`, `lib/memory/memory-service.ts`, `lib/memory/memory-gate.ts`, `lib/mount-index/document-search.ts`, `lib/mount-index/mount-chunk-cache.ts`, `lib/scriptorium/conversation-search.ts`, `lib/help-search.ts`, `lib/tools/memory-dedup.ts`, `lib/database/repositories/{doc-mount-chunks,conversation-chunks,help-docs,vector-indices}.repository.ts`, `lib/background-jobs/handlers/{embedding-generate,embedding-reindex}.ts`, `app/api/v1/mount-points/route.ts`, `app/api/v1/mount-points/[id]/route.ts`, `migrations/scripts/normalize-embeddings-unit-vectors.ts`, `migrations/scripts/index.ts`)

#### Features

- **Aurora & Scriptorium — Characters always carry their own vault into the room**: When an LLM takes the stage as a character and reaches for the `doc_*` tools, that character's own document vault is now extended to it as a matter of course — even when the vault has not been independently linked to the active project. The path resolver now considers two sources when deciding which mount points the tools may see: the stores linked to the active project (as before) and the single store named by `characters.characterDocumentMountPointId` (new). The two sets are merged and deduplicated, so linking a vault to a project on top of the character link does not multiply it. `DocEditToolContext` and `PathResolutionContext` each grew an optional `characterId`, threaded in from the chat `ToolExecutionContext`; `getAccessibleMountPoints()` and `resolveDocumentStorePath()` consult it. To plug the matching gap on the other end, character creation no longer leaves new characters without a vault until the next server boot: the provisioning logic previously inlined in `backfill-character-vaults.ts` has been extracted into a reusable `ensureCharacterVault()` helper at `lib/mount-index/character-vault.ts`, and the three character-create paths (`POST /api/v1/characters`, `?action=quick-create`, `?action=import`) now call it fire-and-forget immediately after `repos.characters.create()`. The helper remains idempotent — if a character already carries `characterDocumentMountPointId`, it is a no-op — so the startup backfill continues to catch anything a failed background provisioning dropped on the floor. (`lib/doc-edit/path-resolver.ts`, `lib/tools/handlers/doc-edit-handler.ts`, `lib/chat/tool-executor.ts`, `lib/mount-index/character-vault.ts`, `lib/startup/backfill-character-vaults.ts`, `app/api/v1/characters/route.ts`)
- **Scriptorium — Startup backfill of character vaults**: On every server boot, Quilltap now sweeps through every Aurora character that isn't already linked to a character document store and conjures one in the Scriptorium on its behalf. The new vault is a database-backed store with `storeType='character'`, named after the character (e.g. a character called *Friday* acquires *Friday Character Vault*), scaffolded through the existing `scaffoldCharacterMount()` helper, and then populated from the character's current data: `identity.md` gets a name-and-pronouns-and-title-and-aliases header, `description.md` / `personality.md` / `example-dialogues.md` receive the corresponding character fields verbatim, `physical-description.md` renders each entry from `physicalDescriptions[]` as a Markdown section with all its prompt fields, `properties.json` serializes `pronouns`, `aliases`, `title`, `firstMessage`, and `talkativeness`, and `wardrobe.json` captures the character's `WardrobeItem[]` / `OutfitPreset[]` from the DB plus any legacy `clothingRecords[]` migrated in as synthetic items (with `migratedFromClothingRecordId` set so the provenance is preserved). Named `systemPrompts[]` and `scenarios[]` each get their own file in `Prompts/` and `Scenarios/` respectively. The backfill runs as new Phase 3.2 in `instrumentation.ts` — after file-storage init and before the mount-point scan, so freshly created vaults land in the first scan pass. It is idempotent (characters with `characterDocumentMountPointId` already set are skipped) and fault-tolerant (a failure for one character logs and continues to the next). The scaffold itself has been extended so new empty vaults match the populated shape: `example-dialogues.md` is now part of the preset structure, and `properties.json` gains `firstMessage: ''` and `talkativeness: 0.5` (with `pronouns` flipped from empty-string to `null` to keep the schema object-shaped). (`lib/startup/backfill-character-vaults.ts`, `lib/mount-index/character-scaffold.ts`, `instrumentation.ts`, `help/mount-points.md`)
- **Aurora — Link a character to its character document store**: Characters now carry an optional `characterDocumentMountPointId` field that names a single Scriptorium document store (by convention, one classified as *Character* and backed by the database). This lays the first stone for the larger migration in which everything about a character except memories — identity, description, personality, scenarios, prompts, wardrobe, lore, images — eventually lives inside a dedicated store rather than in character columns. The field is a soft foreign key (nullable, no FK constraint, since the mount point lives in a separate database) wired through the character Zod schema, the characters table (via migration `add-character-document-mount-point-field-v1`), the create/update API validation, and `.qtap` export/import. On import, if the referenced mount point isn't part of the same archive, the link is cleared rather than left dangling; if it is, it's remapped to the newly-created mount point's id. No UI surface yet — a dropdown in the character edit dialog will follow once the broader character-in-store plan firms up. (`lib/schemas/character.types.ts`, `migrations/scripts/add-character-document-mount-point-field.ts`, `app/api/v1/characters/route.ts`, `app/api/v1/characters/[id]/route.ts`, `lib/import/quilltap-import-service.ts`, `docs/developer/DDL.md`)
- **Scriptorium — Character document stores arrive pre-scaffolded**: When a database-backed document store is created with `storeType='character'`, or when an existing database-backed store is flipped to Character via the Edit dialog, Quilltap now lays in the conventional preset structure automatically: four blank Markdown files (`identity.md`, `description.md`, `personality.md`, `physical-description.md`), two seeded JSON files (`properties.json` with `{pronouns, aliases, title}` mirroring the Character schema shape, and `wardrobe.json` with empty `items[]` / `presets[]` and a slot-null `outfit` mirroring `lib/schemas/wardrobe.types.ts`), and five empty folders (`Prompts`, `Scenarios`, `lore`, `images`, `files`). Filesystem and Obsidian character stores are left untouched — the user is presumed to be pointing at an existing vault of their own. The scaffold is idempotent: pre-existing files are never overwritten, so flipping a populated store to Character simply fills in whatever gaps remain. Implementation lives in a new `lib/mount-index/character-scaffold.ts` helper and is called from both `POST /api/v1/mount-points` and the `PATCH` endpoint's storeType-flip branch. (`lib/mount-index/character-scaffold.ts`, `app/api/v1/mount-points/route.ts`, `app/api/v1/mount-points/[id]/route.ts`, `help/mount-points.md`)
- **Scriptorium — `storeType` classification on document stores**: Every document store now carries a `storeType` field in addition to its `mountType`. The two allowed values are `'documents'` (the default — general notes, references, research) and `'character'` (character sheets and related Aurora material). The Create and Edit dialogs expose a *Contents* dropdown so the classification can be set at creation and changed at any time afterwards. The card on the Scriptorium index shows a new *Documents* / *Character* badge. The field is wired through the API (`POST`/`PATCH /api/v1/mount-points[/:id]`), `.qtap` export/import (backward-compatible; older archives default to `'documents'`), the public export JSON schema, and `DDL.md`. The `doc_mount_points` table gains the new column via in-repo idempotent `ALTER TABLE ADD COLUMN` on first access for existing installations.
- **Scriptorium — Arbitrary binary uploads in database-backed stores with extracted-text embeddings**: Any file type can now be uploaded to a database-backed document store. PDF and DOCX uploads get their plain text extracted into a new `extractedText` column on `doc_mount_blobs`; that text is chunked, embedded, and made searchable alongside native `.md` / `.txt` / `.json` documents. Arbitrary binaries (zip, audio, video, `.bin`, etc.) are stored as-is and surface in the tree with `fileType='blob'` — future converters can fill in extracted text for additional formats without schema changes. Bitmap uploads (PNG/JPEG/GIF/TIFF/HEIC/HEIF/AVIF) still transcode to WebP; already-WebP uploads are now stored as-is without re-encoding. The Scriptorium detail page's separate BlobManager pane has been folded into the main file table — every file, text or binary, lives in one unified tree with an Upload button on database-backed stores. `doc_read_file` on a blob with extracted text returns the derived plain text with `derivedFromBlob: true` so LLMs can read PDFs and DOCX uploads through the same tool they use for Markdown. Folder moves now also rename contained blobs, closing the last gap from the earlier blob-path TODO. `.qtap` export and import round-trip the new blob columns (`extractedText`, `extractedTextSha256`, `extractionStatus`, `extractionError`, `descriptionUpdatedAt`); the public export schema has been updated to match. Older `.qtap` archives that predate these fields import without regressions and leave new blobs in the default `extractionStatus='none'` state.

- **Scriptorium — Convert and Deconvert document stores between backends**
  - Each document-store card on the Scriptorium page now sports a **Convert** button (on filesystem and Obsidian stores) and a **Deconvert** button (on database-backed stores) alongside the familiar Scan/Edit/Delete controls.
  - **Convert** reads every indexed file from the store's `basePath` and moves its bytes inside the encrypted mount-index database: text files land in `doc_mount_documents`, binaries in `doc_mount_blobs`. The original files on disk are left untouched. The store's `mountType` flips to `'database'` and the filesystem watcher detaches.
  - **Deconvert** prompts for a target directory (which must be empty or nonexistent), writes every document and blob out to disk at its relative path, flips `mountType` back to `'filesystem'`, and attaches a fresh filesystem watcher against the new `basePath`.
  - **Embeddings are preserved** across either direction. The `doc_mount_files` row and its `doc_mount_chunks` children (including the `embedding` BLOB) stay in place throughout; only the `source` column flips. No re-embedding is necessary.
  - New `POST /api/v1/mount-points/:id?action=convert` and `?action=deconvert` endpoints for programmatic access; the deconvert body accepts `{ targetPath }` and validates emptiness server-side.
  - New `conversionStatus` / `conversionError` columns on `doc_mount_points` drive the in-flight "Converting…" / "Deconverting…" badges on each card; an in-repo ALTER-TABLE migration adds them automatically on first access after upgrade.
- **Scriptorium — Database-backed document stores and universal blob layer**
  - New `mountType: 'database'` option for Scriptorium document stores. Documents and binary assets live entirely inside the SQLCipher-encrypted `quilltap-mount-index.db` — no filesystem path required. DB-backed stores are automatically covered by the existing 24-hour physical-backup sweep.
  - New universal blob table (`doc_mount_blobs`) available to every mount type. Uploaded images are transcoded to WebP server-side via `sharp`; original filename, original MIME type, user-supplied description, and SHA256 are preserved as metadata.
  - New blob REST endpoints: `GET`/`POST /api/v1/mount-points/:id/blobs` (list / upload multipart) and `GET`/`PATCH`/`DELETE /api/v1/mount-points/:id/blobs/<path>` (stream / update description / delete).
  - Four new `doc_*` tools — `doc_write_blob`, `doc_read_blob`, `doc_list_blobs`, `doc_delete_blob` — let characters upload, reference, and curate images alongside the existing document editing tools.
  - Scriptorium UI gains a third "Database-backed" mount type option and a BlobManager on every store detail page (thumbnail grid, description editing, "Copy Markdown" helper).
  - `MessageContent` renderer accepts a `blobMountPointId` prop; when set, relative Markdown image references like `![alt](images/avatar.webp)` resolve through the blob API so DB-backed assets display inline.
  - `.qtap` export/import gains a new `document-stores` entity type that round-trips mount point configuration plus — for DB-backed mounts — document bodies and base64-encoded blobs.

- **Memory search dimension mismatch fallback**: When the search embedding profile produces different dimensions than the stored vector index, `searchMemoriesSemantic` now detects the mismatch before calling vector search and falls back to text-based search instead of silently returning zero results
- **Text search fallback broadened to per-word matching**: The text-based memory search fallback now searches for individual significant words (filtering stop words) when the full query phrase doesn't match, so multi-word queries find relevant memories even without exact substring matches
- **Removed per-chat embedding profile override**: The `embeddingProfileId` field in `cheapLLMSettings` was removed — the system now always uses the single default embedding profile for all search operations, preventing mismatches between the profile used to build the vector index and the one used at search time
- **Tasks Queue UI**: The paused-jobs count was never shown in the queue stats panel because the API response omitted the `paused` field — it is now included.
- **Tasks Queue UI**: Jobs of type `SCENE_STATE_TRACKING`, `CHARACTER_AVATAR_GENERATION`, and `CONVERSATION_RENDER` appeared as raw type identifiers instead of human-readable names; they now display correctly.
- **Tasks Queue UI**: Non-LLM background jobs (embedding generation, vocabulary refit, re-index, avatar generation, conversation render, and story background generation) were incorrectly contributing 500 estimated tokens each to the queue token estimate; they now correctly contribute 0.
- **Tasks Queue UI**: The `ProcessorStatus` type now includes `embeddingInFlight` and `embeddingConcurrency` to match what the API actually returns.
- **Embedding model swap dimension mismatch**: Full re-embed (EMBEDDING_REINDEX_ALL) now clears each character's vector index before enqueuing new jobs, so switching to a model with different dimensions (e.g. 1536 → 4096) no longer fails every job with "Vector dimension mismatch"
- **Stale embedding jobs on re-embed**: Re-embed now cancels all pending/failed/orphaned EMBEDDING_GENERATE jobs from the previous run before enqueuing fresh ones, preventing zombie jobs from competing for processor slots
- **Slow re-embed enqueue**: Reindex handler now batch-inserts all embedding jobs in chunks of 200 (single SQLite transactions) instead of 3 sequential DB calls per entity — enqueue phase drops from minutes to seconds for large instances
- **Orphaned PROCESSING jobs blocking new work**: On startup, all PROCESSING jobs are now killed (DEAD) instead of reset to PENDING, since no job can legitimately be mid-flight at server start. Prevents stale retries from blocking fresh batch-inserted jobs. Embedding concurrency also guards against over-claiming via early bail when slots are full, with proactive back-fill when a slot frees.
- **Embedding BLOB Registration**: Fixed race condition where memory embeddings could be stored as JSON text instead of Float32 BLOBs, causing dimension mismatches during vector search. BLOB columns for `memories`, `vector_entries`, and `conversation_chunks` are now registered at database initialization time rather than lazily in individual repositories.
