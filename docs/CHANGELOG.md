# Quilltap Changelog

## Recent Changes

### 3.3-dev

- feat: Phase 3 theme registry system — Ed25519 crypto module (`lib/themes/crypto.ts`) for key generation, signing, and verification of registries and bundles using Node.js built-in `crypto`; registry client (`lib/themes/registry-client.ts`) for managing remote theme registries with source add/remove/toggle, registry index fetching with 1-hour cache TTL, Ed25519 signature verification, SHA-256 hash verification on downloads, search across registries, and update checking; new API actions on `/api/v1/themes`: `registry` (browse), `registry-sources` (list sources), `updates` (check updates), `add-source`, `remove-source`, `refresh`, `install-registry`; new `ThemeBrowser` UI component in Appearance settings with collapsible registry browser, search, install from registry, source management with add/remove forms, verified/unverified badges, and compatibility indicators; CLI `quilltap themes registry` subcommands for list/add/remove/refresh/keygen/sign; CLI `quilltap themes search` and `quilltap themes update` commands; registry index format with Zod schemas (`RegistryIndexSchema`, `RegistrySourceSchema`, `RegistryThemeSchema`); bundle signature format for integrity verification; help documentation updated with registry CLI commands and UI browsing guide
- feat: Phase 2 theme bundle infrastructure — `quilltap themes` CLI subcommand with list/install/uninstall/validate/export/create commands; standalone theme validation module for CLI use without Next.js dependencies; `create-quilltap-theme` v2.0.0 now defaults to bundle format (use `--plugin` for legacy npm plugin); all 5 bundled themes (Art Deco, Earl Grey, Great Estate, Old School, Rains) converted to `.qtap-theme` bundle directories in `themes/bundled/` and loaded at startup alongside user-installed bundles; npm plugin themes marked as deprecated throughout the system with deprecation badges in theme selector UI; theme registry loads bundled themes from app source directory before user themes; help documentation updated with CLI commands
- feat: Declarative `.qtap-theme` bundle format for theme distribution — themes can now be packaged as logic-free zip archives containing JSON tokens, CSS overrides, and font files without requiring npm, esbuild, or TypeScript; new `QtapThemeManifestSchema` (Zod) and `public/schemas/qtap-theme.schema.json` for validation; `lib/themes/bundle-loader.ts` handles validation (blocked extensions, size limits, zip bomb protection, path traversal prevention), installation to `<base>/themes/`, uninstallation, and loading; theme registry extended with `source` field (`'plugin' | 'bundle' | 'default'`), `bundlePath`, `registerBundleTheme()`, and `unregisterTheme()` for hot-load/unload; asset and font serving routes updated to resolve `bundle:` prefixed paths from bundle directories; API routes added: `POST /api/v1/themes?action=install` (multipart upload), `POST ?action=install-from-url`, `DELETE /api/v1/themes/:id` (bundle only), `GET ?action=export` (any theme); theme selector UI updated with "Install Theme" button, source badges ("Built-in"/"Plugin"/"Bundle"), uninstall button for bundles, and export option; help documentation updated; uses `yauzl` for secure zip extraction
- fix: Chat UI shows phantom "waiting for response" state after a character finishes replying — `waitingForResponse` was never cleared in the `onDone` happy path of SSE streaming, so if the done event arrived without a prior content chunk the typing indicator and participant sidebar active glow would persist indefinitely; now explicitly reset in both the completion handler and the `finally` block

### 3.2.1

- fix: Release workflow tag filter used regex syntax (`+`) invalid in GitHub Actions glob patterns — replaced with correct glob pattern and added `validate-tag` job with proper regex validation; all build jobs now gate on tag validation; made Discord notification job non-blocking (`continue-on-error`); Linux Electron build now waits for rootfs like other platforms
- fix: `npx quilltap` fails with "sqlite not accessible after retries" because npm hoisting resolves the `better-sqlite3` alias to plain `better-sqlite3` instead of `better-sqlite3-multiple-ciphers` (SQLCipher) — the CLI package now depends on `better-sqlite3-multiple-ciphers` by its real name, resolves it explicitly in `ensureNativeModules()` and `linkNativeModules()`, and symlinks it as `better-sqlite3` for the standalone server; the `quilltap db` subcommand also prefers the SQLCipher build
- fix: Migration retry loop silently swallows all database connection errors — `waitForDatabaseReady()` now logs the actual error on each attempt so failures are diagnosable
- fix: `getSQLiteDatabase()` caches a half-initialized connection when pragma setup fails — subsequent retries return the same broken instance instead of retrying; now closes the connection on error so retries start fresh

- fix: Remove Ben as default seed character, keep only Lorian & Riya — deleted `first-startup/characters/ben.json` and fixed early return in `seedInitialData()` that prevented `.qtap` imports from running when no JSON seed characters existed
- fix: Setup wizard selectable options (embedding provider, image provider, provider selection, profile archetype) all looked selected simultaneously due to undefined `qt-bg-active`/`qt-bg-hover` classes — new `qt-option-selected` and `qt-option-unselected` utility classes in `_utilities.css` provide clear visual distinction with primary-tinted selected state and transparent unselected state with hover feedback; replaced all phantom class usage across 6 wizard components
- fix: API keys left as encrypted ciphertext after the `drop-api-key-encryption-columns` migration are now detected and decrypted on startup; keys that cannot be recovered (iv/authTag columns already dropped) trigger a user-facing toast notification advising re-entry in Settings; new migration `decrypt-api-key-values-v1`, migration warning infrastructure in startup state, `/api/v1/system/migration-warnings` endpoint, and `MigrationWarningNotifier` frontend component
- docs: Add developer documentation for database encryption system — new `docs/DATABASE_ENCRYPTION.md` covers SQLCipher architecture, `.dbkey` file format, pepper key management, startup lifecycle state machine, plaintext-to-encrypted conversion, unlock API endpoints, passphrase management, related migrations, CLI access, and security details; updated `docs/DATABASE_ABSTRACTION.md` with encryption section; updated `docs/BACKUP-RESTORE.md` to replace outdated `ENCRYPTION_MASTER_PEPPER` references with `.dbkey` file documentation and `npx quilltap db` commands
- feat: Change encryption passphrase from Settings — new "Encryption Passphrase" card on the Data & System tab lets users change, add, or remove the passphrase that protects the `.dbkey` encryption key file; re-wraps the pepper with new PBKDF2 parameters without re-encrypting the database; updates both main and LLM logs `.dbkey` files atomically; new `changePassphrase()` function in `lib/startup/dbkey.ts` and `?action=change-passphrase` endpoint on `/api/v1/system/unlock`
- security: Remove `hasPassphrase` flag from `.dbkey` file — the flag explicitly revealed whether a user passphrase protected the encryption key, leaking information to attackers with filesystem access; provisioning now always tries the internal passphrase first and falls back to prompting the user; existing `.dbkey` files are automatically rewritten on read to strip the legacy field; CLI `quilltap db` subcommand updated with matching try-first decryption logic
- fix: Database encryption converter fails with "database is locked" on iCloud-synced directories — the converter now works on a temporary copy (`quilltap.db.encrypting`) instead of the original file, avoiding file coordination locks held by iCloud sync, Spotlight, or other macOS services; also encrypts databases immediately after first-run setup so there is no plaintext window between setup and next restart; closes existing DB connections before conversion during deferred startup
- fix: Passphrase unlock fails when migrating from legacy pepper vault to `.dbkey` file — when a user had a passphrase set in the old `pepper_vault` but no `.dbkey` file existed yet, the unlock endpoint rejected the attempt because the dbkey module's internal state was `needs-setup` instead of `needs-passphrase`; the unlock handler now detects the legacy scenario and routes through `unlockPepper()`, then automatically migrates the pepper to a `.dbkey` file
- feat: SQLCipher database encryption at rest — databases are now encrypted using SQLCipher; `.dbkey` file replaces in-DB pepper vault; field-level API key encryption removed (whole DB encrypted); locked mode for passphrase-protected keys; `quilltap db` CLI subcommand for querying encrypted databases
- fix: Database encryption converter used unavailable `sqlcipher_export` function and failed in WAL journal mode — replaced with `PRAGMA rekey` (in-place encryption) after switching to DELETE journal mode; restores WAL mode after rekey
- fix: Multi-character assistant prefill trailing whitespace — the `[CharacterName]` identity anchor appended a trailing space, which Anthropic's API rejects with "final assistant content cannot end with trailing whitespace"; removed the trailing space
- docs: Add feature plan for OpenAI Responses API migration (`features/openai-responses-api.md`) — documents phased approach to migrate `qtap-plugin-openai` from Chat Completions to Responses API, covering core migration, conversation chaining via `previous_response_id`, and enhanced server-side tools; references existing Grok plugin as implementation template
- fix: Multi-character identity confusion — in multi-character chats, weaker LLMs could respond as the wrong character despite correct system prompts; now appends an assistant prefill message (`[CharacterName]` + space) to anchor the model's response identity, leveraging the existing `stripCharacterNamePrefix()` cleanup
- fix: LLM Inspector logs no longer truncate message content to 500 chars — replaced the dual `contentPreview`/`fullMessages` structure with full content storage directly in the `messages` array; the UI still shows first 500 chars with an expandable "Show all" button; backward-compatible with existing log entries via fallback field reads
- fix: LLM log entries for chat messages now populate `characterId`, making it possible to trace which character each log entry belongs to
- feat: Inject character pronouns into Commonplace Book memory extraction prompts — when a character has pronouns set (e.g., she/her/her), they are now appended after the character name in all cheap LLM memory extraction prompts (user memory, character memory, and inter-character memory), preventing pronoun confusion during extraction; pronouns appear in the PARTICIPANTS context block, conversation transcript labels, TARGET CHARACTER lines, and inter-character observer/subject headers; new shared `formatNameWithPronouns()` utility in `lib/memory/format-utils.ts`
- feat: Memory weighting system for the Commonplace Book — new `calculateEffectiveWeight()` combines base importance with exponential time decay (30-day half-life, configurable importance floor) using `max(createdAt, lastReinforcedAt)` as the reference time so passive retrieval never resets the decay timer; integrated into semantic search ranking (60% cosine similarity + 40% effective weight), context injection sorting (weight-primary with score tiebreaker), and housekeeping hard-cap enforcement (replaces ad-hoc 4-factor composite score); memories injected into LLM context now include relative age labels (`[yesterday]`, `[3 weeks ago]`, `[2 months ago]`) so the LLM can distinguish recent from ancient memories; removed lastAccessedAt feedback loop from retrieval path
- feat: Thread `chatId` through all cheap LLM task functions for LLM Inspector visibility — all 18 exported functions in `cheap-llm-tasks.ts` (memory extraction, title generation, summarization, compression, image prompt crafting, scene context derivation, story background prompts, appearance resolution, keyword extraction, attachment description) now pass `chatId` to `executeCheapLLMTask` so their calls appear in the per-chat LLM Inspector panel; added `logLLMCall()` to the Concierge moderation provider path so OpenAI moderation API calls are also logged; updated all call sites across orchestrator, context summary, compression, background job handlers, image generation handler, and API routes
- fix: User messages no longer vanish from the chat UI when the LLM provider returns a streaming error (e.g., Anthropic `overloaded_error`) — the backend already persists the user message before streaming begins, but the frontend was removing the optimistic message on error; now re-fetches the chat to sync with the saved server state instead
- chore: Remove unused eslint-disable directives for `quilltap/no-quilltap-misspelling` in Electron files
- feat: LLM Inspector Panel — new right-edge slide-over panel accessible from the chat toolbar (terminal icon) or via Cmd+Shift+L / Ctrl+Shift+L that shows all LLM activity for a chat (chat messages, tool continuations, memory extraction, title generation, danger classification, context compression, and more) in chronological order with collapsible detail cards, type-colored badges, client-side filtering by category, and scroll-to-entry when opened from a per-message "View LLM logs" button; includes new reusable `SlideOverPanel` component, combined backend query (`findAllForChat`) that fetches logs by both chatId and messageId association, and new `qt-slide-over-*` / `qt-inspector-*` CSS classes for full theme compatibility; replaces the old `LLMLogViewerModal` in the chat page (modal is preserved for Settings and Character pages)
