# Quilltap Changelog

## Recent Changes

### 4.4-dev

#### Per-character Librarian summaries

- **Summaries are now whispered, not broadcast.** The Librarian still regenerates `chat.contextSummary` at the same checkpoints (used by title generation, danger classification, story background, etc.), but the conversation-summary message in the transcript is now produced once per LLM-controlled CHARACTER participant and whispered privately to them via `targetParticipantIds`. Each character may have entered mid-chat or stepped away and back, so each gets a summary tailored to what they actually saw.
- **Visible-slice computation.** For each participant, the summary input is built by composing (1) the existing `filterMessagesByHistoryAccess` slice (skip messages before they joined unless `hasHistoryAccess === true`), (2) a new presence-window slice that drops messages during periods they were `absent` or `removed`, and (3) the existing `filterWhisperMessages` step. Presence windows are computed from Host status announcements — see the new `computePresenceWindowsForParticipant` and `filterMessagesByPresenceWindows` helpers in `lib/chat/context/message-attribution.ts`.
- **Host status announcements now carry structured data.** Added `hostEvent` JSON column on `chat_messages` (`{ participantId, toStatus }`) plus a Zod field on `MessageEventSchema` and `ChatMessageRowSchema`. `postHostAddAnnouncement` / `postHostRemoveAnnouncement` / `postHostStatusChangeAnnouncement` in `lib/services/host-notifications/writer.ts` now require `participantId` and persist `hostEvent`. Migration: `migrations/scripts/add-host-event-field.ts` (`add-host-event-field-v1`). Other Host announcements (scenario, roster, timestamp, silent-mode, join-scenario) leave `hostEvent` NULL.
- **Call sites updated** in `app/api/v1/chats/route.ts`, `app/api/v1/chats/[id]/helpers.ts`, and `app/api/v1/chats/[id]/actions/participants.ts` to pass `participantId` (and `initialStatus` for adds). The add path was reordered so the new participant is resolved before the Host announcement fires.
- **Old whispered summaries are swept.** When a fresh per-character summary whisper is posted, prior Librarian summary whispers targeted at the same participant are deleted from the chat. Identified by `systemSender === 'librarian'`, exact single-target match, and the new `SUMMARY_CONTENT_PREFIX` content marker. Old broadcast summaries (from before this change) are not touched.
- **New repository helper.** `chats.repository.ts` and `lib/repositories/user-scoped.ts` expose `deleteMessagesByIds(chatId, ids[])`; underlying op added on `chats-messages.ops.ts`.
- **Reuse-when-equal optimization.** When a participant's visible slice contains exactly the same message IDs that fed the global summary (single-character chats with no absences and no whispers, etc.), the global summary text is reused verbatim rather than calling the cheap LLM again.
- **Catch-up summary on join.** When a character is added to a chat with `hasHistoryAccess === true`, `triggerCatchUpSummaryAsync` fires in the background to whisper a summary of prior history to the joining character right away — they don't have to wait for the next checkpoint. No-op when `hasHistoryAccess` is false; that case waits for the next regular regen.
- **Writer signature change.** `postLibrarianSummaryAnnouncement` now accepts `targetParticipantIds`. The exported `SUMMARY_CONTENT_PREFIX` constant is the canonical body marker for sweeping prior summaries.

#### System Prompt Refactor (Phase A + B + C + D + E + F + G + H + I)

Begun the move of dynamic state out of the per-turn system prompt and into Staff-authored ASSISTANT messages, so the system prompt can become a static, cache-friendly artifact and the chat transcript can become the single source of truth for state changes.

- **New Staff persona — the Commonplace Book.** Added `commonplaceBook` to the `systemSender` enum in `lib/schemas/chat.types.ts`, `lib/database/repositories/chats-messages.ops.ts`, `app/salon/[id]/types.ts`, and `public/schemas/qtap-export.schema.json`. New writer module at `lib/services/commonplace-notifications/writer.ts` exports `postCommonplaceWhisper` (persistent path) and `voiceCommonplaceContent` (per-turn voicing helper). Avatar at `public/images/avatars/commonplace-book-avatar.webp` wired into `getMessageAvatar` in `app/salon/[id]/page.tsx`. No database migration needed — the `chat_messages.systemSender` column has no CHECK constraint.
- **Memory tail moved out of the system prompt.** The memory recap, relevant memories, and inter-character memories used to be concatenated onto the system prompt. They are now persisted as a single consolidated Commonplace Book whisper per turn — voiced in the steampunk Quilltap style ("*The Commonplace Book lays open at your bookmark…*") — visible in the salon UI right after the user's message that triggered it, with the Commonplace Book avatar. For the LLM call, the same recall content is inlined into the new user message body in plain second-person framing ("You remember the following entries that bear on this moment: …") — no Staff persona, no meta-narrative, no consecutive-assistant alternation issue. Past Commonplace Book whispers are filtered out of `existingMessages` before context building (they live in the transcript for the user, but the LLM gets fresh recall each turn). The memories table remains the source of truth.
- **System transparency filter no longer drops Staff messages.** For opaque characters (`systemTransparency !== true`), `lib/services/chat-message/context-builder.service.ts` now strips the `systemSender` attribution but keeps the message body, so the LLM still receives content (scenario, status, memory) as a generic assistant line. Previously, opaque characters lost the message entirely. The salon UI is unaffected — the human user always sees Staff-attributed messages with their avatars.
- **Host content moved out of the system prompt (Phase C wire-up).** Scenario, user-character introduction, multi-character roster, own-status reminder, silent-mode rule, recent status changes, and join-scenario context now ship as Host-authored ASSISTANT messages in the chat transcript instead of being concatenated onto the per-turn system prompt.
  - **At chat creation** (`createInitialMessages` in `app/api/v1/chats/route.ts`): right after the SYSTEM message, the Host posts the scenario announcement (if a scenario is set), the user-character introduction (if a user character is configured), and — for multi-character chats — a welcome announcement for each LLM-controlled CHARACTER participant so each character learns about the others from the transcript.
  - **At participant add** (`processChatUpdates` in `app/api/v1/chats/[id]/helpers.ts` and `handleAddParticipantAction` in `app/api/v1/chats/[id]/actions/participants.ts`): when a new participant has a non-empty `joinScenario` and `hasHistoryAccess === false`, the Host posts a private join-scenario whisper targeted at the joining participant via `targetParticipantIds`. Replaces the per-turn `## How You Entered This Conversation` insertion.
  - **At silent-mode transitions** (`handleParticipantUpdate` in `app/api/v1/chats/[id]/helpers.ts`): when a participant transitions into or out of silent mode, the Host posts a private silent-mode entry or exit whisper carrying the full silent-mode rule directly to the affected character. Replaces the per-turn `## Silent Mode Active` block.
  - **System-prompt blocks dropped.** `lib/chat/context/system-prompt-builder.ts` no longer emits the `## Scenario`, `## User Character`, `## Other Participants in This Conversation`, `## Your Current Status`, `## Silent Mode Active`, or `## Recent Status Changes` blocks. `lib/chat/context-manager.ts` no longer appends `## How You Entered This Conversation` to the per-turn system prompt. The `otherParticipants`, `respondingCharacterStatus`, and `statusChangeNotifications` parameters remain on `buildSystemPrompt` for now to avoid call-site churn; they are unused and will be removed in Phase I cleanup.
  - **Tests updated.** `__tests__/unit/context-management.test.ts` was updated to assert that persona/user-character info and scenario text no longer appear in the per-turn system prompt, and that inter-character memories now ride on the new user message body rather than the system message.
- **Aurora content moved out of the system prompt (Phase D).** Outfit-change notifications, current outfit, and the available wardrobe roster now ship as Aurora-authored ASSISTANT messages in the chat transcript instead of being concatenated onto the per-turn system prompt.
  - **New writer** at `lib/services/aurora-notifications/writer.ts` exports `buildOpeningOutfitContent`, `buildOutfitChangeContent`, `postOpeningOutfitWhisper`, and `postOutfitChangeWhisper`. Aurora whispers do not need the dual persona/LLM voicing pattern that Commonplace Book recall does — the persona-voiced body is acceptable input for the LLM, and for opaque (`systemTransparency=false`) characters the `systemSender` attribution is stripped at context-build time but the body survives, so the content still reaches them as a generic assistant line.
  - **At chat creation** (`createInitialMessages` in `app/api/v1/chats/route.ts`): for each LLM-controlled CHARACTER participant with an equipped outfit, Aurora posts an opening whisper announcing the character's current attire and the alternatives in their wardrobe. Fires after the Host welcomes and before the first ASSISTANT greeting.
  - **On equip/unequip via the sidebar** (`app/api/v1/chats/[id]/actions/outfit.ts`): the `pendingOutfitNotifications` chat-field flow has been retired. Outfit changes now flow exclusively through the debounced Aurora wardrobe announcement (`enqueueWardrobeOutfitAnnouncement` → `handleWardrobeOutfitAnnouncement`). The debounced handler was refactored to use the new `postOutfitChangeWhisper` so its body now includes the available wardrobe alongside the current outfit.
  - **Orchestrator drain dropped.** `lib/services/chat-message/orchestrator.service.ts` no longer reads or clears `chat.pendingOutfitNotifications`. The `outfitChangeNotifications` parameter on the context-builder options remains for now (unused) and will be removed in Phase I cleanup.
  - **System-prompt blocks dropped.** `lib/chat/context/system-prompt-builder.ts` no longer emits the `## ⚠️ Outfit Change Notice`, `## Current Outfit`, or `## Available Wardrobe` sections. The `wardrobeContext` and `outfitChangeNotifications` parameters remain on `buildSystemPrompt` (unused) for the same Phase I cleanup.
  - **Test updated.** `__tests__/unit/app/api/v1/chats/[id]/actions/outfit.test.ts` no longer asserts that the equip handler performs a bulk wardrobe lookup — the lookup moved into the debounced Aurora job.
  - **Deferred follow-ups** (still pending Phase D plumbing): mentioned-characters whisper (needs novelty tracking — schema field or a stable body-marker — to avoid re-announcing the same characters every turn), and physical-state delta whispers (no triggers exist yet for injury/transformation/age-up events).
- **Prospero project context moved out of the system prompt (Phase E).** The `## Project Context` block is no longer concatenated onto the per-turn system prompt. Project description and instructions now ride as Prospero-authored ASSISTANT messages in the chat transcript.
  - **Writer extended** at `lib/services/prospero-notifications/writer.ts` with `buildProjectContextContent` and `postProsperoProjectContextAnnouncement`. The body is persona-voiced ("Prospero opens his ledger to the project at hand…"); for opaque (`systemTransparency=false`) characters the `systemSender` attribution is stripped at context-build time but the body survives.
  - **At chat creation** (`createInitialMessages` in `app/api/v1/chats/route.ts`): when a project is attached and has description or instructions, Prospero posts a project-context whisper right after the SYSTEM message and before the Host announcements.
  - **Cadence-based refresh** (`lib/services/chat-message/orchestrator.service.ts`): the previous per-turn `projectContext` injection has been replaced with a Prospero whisper post. When the message count is a non-zero multiple of `projectContextReinjectInterval` (default 5), a fresh whisper is posted into the transcript so the LLM keeps the project's particulars in mind. The just-posted whisper is pushed onto the in-memory `existingMessages` array so it is included in this turn's context.
  - **System-prompt block dropped.** `lib/chat/context/system-prompt-builder.ts` no longer renders the `## Project Context` / `### Project Instructions` block. The `projectContext` parameter on `buildSystemPrompt` is now unused (kept on the signature for Phase I cleanup) and the orchestrator passes `null` for it.
- **Librarian conversation summary moved out of the system prompt (Phase F).** The `## Previous Conversation Summary` block is no longer concatenated onto the per-turn system prompt. Whenever the cheap-LLM summarizer regenerates `chat.contextSummary`, the Librarian now deposits the fresh summary into the chat transcript as a synthetic ASSISTANT message, and the LLM picks it up from conversation history.
  - **Writer extended** at `lib/services/librarian-notifications/writer.ts` with `buildSummaryContent` and `postLibrarianSummaryAnnouncement`.
  - **Wire-up** at `lib/chat/context-summary.ts`: after `repos.chats.update` writes the new summary and the existing `context-summary` event is appended, a Librarian summary whisper is posted with the same body. The chat-level `contextSummary` field is still maintained for other consumers (title generation, scene-state tracking, danger classification).
  - **System-prompt append dropped.** `lib/chat/context-manager.ts` no longer formats the chat's `contextSummary` via `formatSummaryForContext` and concatenates it onto `fullSystemContent`. The per-turn `summaryTokens` budget contribution is now zero.
  - **Deferred follow-ups** (still within Phase F's eventual scope): supersession of older Librarian summary whispers when a new one is posted (currently every regenerate adds a fresh whisper, leaving older ones in the transcript), and migrating `applyContextCompression` from the system-prompt prepend (`buildCompressedSystemMessage`) to a Librarian record-keeping whisper. - **Timestamp moved out of the system prompt (Phase G).** When `timestampConfig.autoPrepend` is set and `shouldInjectTimestamp` returns true (`START_ONLY` + first user turn, or `EVERY_MESSAGE`), the Host now narrates the time as a transcript-resident ASSISTANT message ("The Host marks the time as …") instead of the auto-prepended `Current time: …` line that previously sat at the top of the per-turn system prompt.
  - **Writer extended** at `lib/services/host-notifications/writer.ts` with `buildTimestampContent` and `postHostTimestampAnnouncement`. Using the Host (rather than a `systemSender=null` neutral line as the original plan called for) avoids a salon-UI design call: the existing `getMessageAvatar` would have mis-attributed neutral messages to the first character. The Host's scene-setting role is a natural fit for narrating time. Revisit if a dedicated clock/timekeeper persona is wanted.

The cached body — preamble + base prompt + personality + aliases + pronouns + physical descriptions + example dialogues, with `{{user}}` / `{{scenario}}` / `{{persona}}` resolved — feeds directly into the per-turn `buildSystemPrompt`, which wraps it with the chat-level roleplay template, tool instructions, and tool reinforcement. The LLM provider now sees a stable cache-friendly prefix across turns within a chat session.

Migration: `migrations/scripts/add-compiled-identity-stacks-field.ts` (id `add-compiled-identity-stacks-field-v1`), registered in `migrations/scripts/index.ts`. The order of the per-turn additions was also adjusted so the static identity stack leads, with the dynamic roleplay template / tool instructions / tool reinforcement appended after it — this puts the stable bytes up front and the changing bytes at the end, maximizing provider prompt-cache hits.
  - **Compile triggers.** `compileAllIdentityStacks` runs in `app/api/v1/chats/route.ts` `handleCreate` after `repos.chats.create`. `compileIdentityStackForParticipant` runs in `app/api/v1/chats/[id]/helpers.ts` (after `handleAddParticipant` succeeds in `processChatUpdates`, and after `selectedSystemPromptId` changes via `handleParticipantUpdate`) and in `app/api/v1/chats/[id]/actions/participants.ts` (after the action-dispatch add path and the soft-deleted reactivation path). On `controlledBy` flip (a participant becoming user-controlled or LLM-controlled) the whole map is recompiled because that affects `{{user}}` / `{{persona}}` for every other participant.
  - **Read path.** `lib/chat/context-manager.ts` looks up the cached stack for `respondingParticipant` via `getCompiledIdentityStack` and forwards it as `precompiledIdentityStack` to `buildSystemPrompt`. To make the cache lookup work for single-character chats too, the orchestrator → context-builder → context-manager pipeline now passes `respondingParticipant: characterParticipant` always (previously gated on `isMultiCharacter`).
  - **Known limitation.** Edits to the underlying character record (name, personality, aliases, pronouns, physical descriptions, example dialogues, `systemPrompts`) do NOT auto-invalidate cached stacks across chats — that fan-out is its own design pass. Until it lands, users editing a character mid-chat will keep seeing the stack from compile time. Workaround: read-through fallback ensures correctness when the cached entry is null/missing for a participant, so a future "rehydrate this chat" admin button can simply clear the map.
  - **Tests** added at `__tests__/unit/context-management.test.ts` covering the cache-hit path (precompiled stack used verbatim, character body NOT rebuilt), the cache-miss/fallback path (empty stack triggers fresh build), and the new ordering (identity stack → roleplay template → tool instructions → tool reinforcement).

### 4.3.1

#### Bug Fixes

- **SQLite journal mode default changed from WAL to TRUNCATE**: Quilltap data directories are commonly placed inside cloud-synced folders (iCloud Drive, Dropbox, OneDrive, Google Drive). WAL mode keeps `.db-wal` and `.db-shm` files alongside the main `.db`; if those files sync out of order with the main database — which all four providers can do, especially on dirty shutdown — the database can be corrupted or lose recent writes on the next open. Switching to `TRUNCATE` keeps the rollback journal in a single auxiliary file that is truncated to zero on every commit, eliminating the multi-file sync hazard. Existing databases auto-migrate on first open after upgrade (SQLite checkpoints any pre-existing WAL into the main file as part of the journal-mode transition). The change applies to all three databases — main (`quilltap.db`), LLM logs (`quilltap-llm-logs.db`), and mount index (`quilltap-mount-index.db`) — plus the meta-table connection used during startup.
- **`SQLITE_WAL_MODE` env var inverted**: Previously `SQLITE_WAL_MODE=false` was the opt-out; now `SQLITE_WAL_MODE=true` is the opt-in. Set this only when the data directory lives on a fast local SSD that is not synced to the cloud.
- **Internal**: `lib/startup/db-encryption-converter.ts` now leaves freshly-encrypted databases in TRUNCATE mode rather than WAL, matching the new resting state.
