# Quilltap Changelog

## Recent Changes

### 4.4-dev

#### LLM cost reduction Phase 4: single broadcast summary + edit-aware invalidation

Phase 4 of the cost-reduction plan, scoped down per direction: drop the per-character summary fan-out entirely; one broadcast Librarian summary is visible to every present character. Witness-set / per-character filtering / structured JSON facts (the original Phase 4) are dropped along with their motivation. Edit-aware invalidation lands as a small hook.

- **Single broadcast summary whisper.** `generatePerCharacterSummaryWhispers` and the per-participant visible-message-slice helpers (`buildVisibleMessagesForParticipant`, `visibleConversation`, `GeneratePerCharacterSummariesParams`) are removed from `lib/chat/context-summary.ts`. `generateContextSummary` now sweeps prior librarian summary whispers anchored to older generations and posts ONE Librarian whisper with `targetParticipantIds: null`. Every present character sees the same text. The `lib/services/dangerous-content/resolver.service` import remains — uncensored fallback still applies to the broadcast summary on dangerous chats.
- **Catch-up summary path removed.** `generateCatchUpSummaryForParticipant` and `triggerCatchUpSummaryAsync` are gone. With broadcast summaries already in chat history, newly-joined participants with `hasHistoryAccess === true` see the existing whisper. If no summary exists yet, the next gate fire produces one for everyone. Removed both call sites: `app/api/v1/chats/[id]/helpers.ts` (PUT update path) and `app/api/v1/chats/[id]/actions/participants.ts` (POST add path).
- **`summaryAnchorMessageIds` on chat metadata.** New `summaryAnchorMessageIds: UUID[]` field on `ChatMetadataSchema` and `ChatMetadataBaseSchema` (`lib/schemas/chat.types.ts`) tracks the conversation message IDs (USER + ASSISTANT) that fed the current `contextSummary`. Migration `add-summary-anchor-message-ids-field-v1` ALTERs `chats`; fresh-install DDL in `sqlite-initial-schema.ts` carries the column natively. DDL.md updated. `generateContextSummary` writes the array alongside the summary text on every fire.
- **Edit-aware summary invalidation hook.** New `invalidateContextSummaryIfMessageCovered(chatId, messageIds)` in `lib/chat/context-summary.ts`. When called with one or more changed message IDs, clears `chat.contextSummary`, resets `summaryAnchorMessageIds`, bumps `compactionGeneration` (so memory caches and whisper anchors refresh in step), and zeros `lastSummaryTurn` / `lastFullRebuildTurn` so the next gate evaluation sees "first-time generation" and fires soft. No-ops when none of the changed IDs were covered (typo fixes on a recently arrived message leave the summary intact). Wired into both PUT (edit) and DELETE (single-message + swipe-group cascade) paths in `app/api/v1/messages/[id]/route.ts`.
- **Test coverage.** New `__tests__/unit/lib/chat/summary-invalidation.test.ts` — 7 tests covering empty-id short-circuit, no-summary skip, no-anchor skip, non-covered-message skip, covered-message invalidation (with generation bump check), one-of-many coverage, and findById-failure resilience.

#### LLM cost reduction Phase 3: stable memory pool + whisper anchoring

Phase 3 of the cost-reduction plan. Replaces the per-turn rank-driven memory injection with a two-pool design that keeps the bulk of recall byte-stable across turns within a compaction generation. Identity reinforcement and persona content are deliberately untouched, so the LLM-visible structure is unchanged except for memory ordering and a smaller dynamic head.

- **Two-pool memory architecture.** New `lib/memory/frozen-archive-cache.ts` returns the top 25 memories by effective weight (importance × time decay) at the current `compactionGeneration`, sorted by `memory.id` ascending so the formatted text is byte-stable across turns. Cache key is `(characterId, compactionGeneration)`; on generation bump the entry recomputes from the `findMostImportant` repository call. `lib/chat/context/memory-injector.ts` adds `formatFrozenMemoryArchive` (uses `memory.summary`, no per-turn re-rank, deterministic order) and `formatDynamicMemoryHead` (top 5 by current-turn rank, `[m_xxxx]` rank tags, hard-capped at 200 tokens — `DYNAMIC_HEAD_TOKEN_BUDGET`). `lib/chat/context-manager.ts:buildContext` now calls both: archive in the cacheable-prefix slot of the LLM-context tail, dynamic head appended after with a "Most relevant memories for this turn:" header. Memories already in the archive are filtered out of the dynamic head so the LLM doesn't see the same entry twice.
- **Removed `maxMemories` option from `BuildContextOptions`.** Replaced by `FROZEN_ARCHIVE_SIZE = 25` and `DYNAMIC_HEAD_DEFAULT_SIZE = 5` constants in the new modules. No external callers used the option (only `context-builder.service.ts` set it to its default of 18); dropped the unused parameter from both call sites.
- **Whisper anchoring with `summaryAnchor` field.** New `MessageEvent.summaryAnchor: { compactionGeneration: number } | null` field on `lib/schemas/chat.types.ts`, mirrored on `lib/database/repositories/chats-messages.ops.ts` and `public/schemas/qtap-export.schema.json`. `postLibrarianSummaryAnnouncement` (`lib/services/librarian-notifications/writer.ts`) accepts a `summaryAnchor` parameter and stamps it on every per-character summary whisper. `generatePerCharacterSummaryWhispers` (`lib/chat/context-summary.ts`) replaces the prior content-prefix sweep with an anchor-based sweep: removes only whispers whose anchor is from an older generation (or whose anchor is missing — the legacy fallback uses the prefix match for unanchored rows). New migration `add-summary-anchor-field-v1` ALTERs `chat_messages` to add the column; fresh-install DDL in `sqlite-initial-schema.ts` carries it natively. DDL.md updated.
- **Cache-determinism CI test.** New `__tests__/unit/cache-determinism/system-prompt.test.ts` exercises `buildSystemPrompt` against a fixed character fixture and asserts (a) byte-identical output across consecutive calls and (b) the SHA-256 hash matches a checked-in golden. Hash drift fails CI; engineers update the golden intentionally with `UPDATE_GOLDEN_PROMPT_HASH=1`.
- **Test coverage.** New `__tests__/unit/lib/chat/context/memory-injector.test.ts` covers the new formatters (deterministic order, byte-stability across calls, summary not body, token-budget truncation, [m_xxxx] tags). New `__tests__/unit/lib/memory/frozen-archive-cache.test.ts` covers cache hit/miss by generation, recompute on generation bump, sorted-by-id output, archive-size cap, and per-character isolation. The character-voice eval scaffold (`__tests__/eval/character-voice/README.md`) is updated to defer the runnable harness to Phase 4 — Phase 3 changes are mechanical enough not to need a regression gate.

#### LLM cost reduction Phase 2: triple-gate summarization

Phase 2 of the cost-reduction plan. Replaces the per-checkpoint summarization trigger with a budget-driven triple-gate so most turns reuse the cached summary unchanged. Independent of any architectural redesign — just a smarter trigger. Title-checkpoint logic is unchanged and runs on its own clock.

- **Triple-gate trigger.** New `evaluateSummarizationGate` in `lib/chat/context-summary.ts` returns `'skip' | 'soft' | 'hard'` based on three thresholds: `T_SOFT_TOKEN_THRESHOLD = 8000` (8K tokens added since last refresh), `T_SOFT_TURN_THRESHOLD = 8` (8 interchanges since last refresh), and `T_HARD_TURN_THRESHOLD = 50` (50 interchanges since last full rebuild). `T_promote` is intentionally deferred to Phase 4 since the layered-summary structure does not exist yet. Most turns return `'skip'` and reuse the existing `chat.contextSummary` unchanged. `checkAndGenerateSummaryIfNeeded` calls the gate after the (unchanged) title-checkpoint logic runs and uses `forceRegenerate: decision === 'hard'` so soft fires take the cheap incremental `updateContextSummary` path while hard fires reset accumulated recursive drift.
- **Compaction-generation tracking.** Four new columns on `chats`: `compactionGeneration` (monotonic counter bumped on every fire — used by Phase 3 to anchor memory pools and per-character whispers), `lastSummaryTurn`, `lastSummaryTokens`, `lastFullRebuildTurn`. Schema in `lib/schemas/chat.types.ts` (added to both `ChatMetadataSchema` and `ChatMetadataBaseSchema`); migration `add-summarization-gate-fields-v1`; fresh-install DDL in `sqlite-initial-schema.ts`; DDL.md updated. `generateContextSummary` now writes all four fields when it persists a new summary.
- **Removed redundant inner gate.** The previous `chatNeedsSummary` early-return inside `generateContextSummary` (which rejected updates on chats with < 100 messages) was incompatible with the upstream gate — the gate would decide to fire and `generateContextSummary` would short-circuit. Removed; the gate is the single decision point now. `chatNeedsSummary` is retained as an exported helper for any external caller that might still want it.
- **Per-character whisper outcome metrics.** The "regenerate" vs "reuse global summary" decision in `generatePerCharacterSummaryWhispers` was logged at DEBUG level; promoted to INFO with a structured `whisperOutcome` field (`'regenerated' | 'reused'`) so reuse fraction is queryable without enabling debug logging. Validates Phase 4 ROI before committing to the redesign.
- **Test coverage.** New `__tests__/unit/lib/chat/summarization-gate.test.ts` exercises the gate decision matrix (skip below threshold, soft on either threshold, hard on 50-turn ceiling, hard preferred over soft when both fire) plus a 50-turn simulation that asserts fire count drops from ~9 (old behavior) to 5–8 with at least one hard fire.

#### LLM cost reduction Phase 1: cache observability + provider parity

Phase 1 of the cost-reduction plan in `docs/developer/features/llm_api_costs_breakdown.md`. Closes provider-parity gaps in cache instrumentation, canonicalizes the tools array, and adds the cache-key parameters that providers need to keep cache hits from sharding across machines. No behavioral changes to chat output.

- **Cache token extraction on OpenAI / Grok / Google plugins.** Anthropic already returned `cacheReadInputTokens`; the others didn't. `plugins/dist/qtap-plugin-openai/provider.ts` and `qtap-plugin-grok/provider.ts` now read `usage.input_tokens_details.cached_tokens` from the Responses API; `qtap-plugin-google/provider.ts` reads `usageMetadata.cachedContentTokenCount`. All three populate `cacheUsage` on `LLMResponse` and the final stream chunk so the existing `streaming.service.ts` → `logLLMCall` path captures hit counts end-to-end. Bumped patch versions on each plugin (manifest + package).
- **Per-tier prefix hashes in `llm_logs`.** New `requestHashes` JSON column on the `llm_logs` table holds SHA-256 (truncated 16-char) hashes of the cacheable tiers: system block 1, system block 2, the canonicalized tools array, and the frozen-history tail (everything except the trailing message). Hash drift in any tier signals the provider cache will miss. Computed in `lib/llm/cache-prefix-hashes.ts:computeRequestPrefixHashes` and threaded through `streaming.service.ts`. New migration `add-llm-logs-request-hashes-column-v1` ALTERs the existing `llm_logs` table; the constant `LLM_LOGS_TABLE_DDL` in `move-llm-logs-to-separate-db.ts` was also extended for fresh installs. Schema added to `lib/schemas/llm-log.types.ts` (`LLMLogRequestHashesSchema`) with re-exports in `lib/schemas/types.ts`. DDL.md updated.
- **Tool list canonicalization.** `lib/tools/canonicalize.ts:canonicalizeUniversalTools` sorts `universalTools` alphabetically by `function.name` and recursively sorts JSON-Schema object keys inside each tool's `parameters`. Applied in `lib/tools/plugin-tool-builder.ts` after the tools are assembled and before the provider's `formatTools()` is called. Without this, plugin tools registered via `Map` iteration order or schema-key authoring order produced byte-different tool arrays across requests, defeating the prefix cache. Snapshot test in `__tests__/unit/lib/tools/canonicalize.test.ts`.
- **Provider `prompt_cache_key` parameters.** New module `lib/llm/cache-key.ts` defines `PROMPT_CACHE_STRUCTURE_VERSION` (currently `1`) and `buildPromptCacheKey(chatId)` → `quilltap:chat:<chatId>:v<version>`. `streaming.service.ts` injects this into `profileParameters.promptCacheKey`. The OpenAI plugin reads it and sets `prompt_cache_key` on Responses API requests (plus `prompt_cache_retention: '24h'` for gpt-5.1 / gpt-5.2 / gpt-5.5 — `gpt-5.3-chat-latest` is intentionally excluded since it's not in the extended-retention model list). The Grok plugin (which targets the xAI Responses API via the OpenAI SDK) sets the same field; without it Grok round-robins across servers and effectively never hits cache. Bump policy: changing tool schemas, system-prompt structure, persona block format, or memory-pool format requires bumping `PROMPT_CACHE_STRUCTURE_VERSION` to invalidate provider caches in lockstep.
- **Anthropic mid-history breakpoint + streamMessage fix.** New `applyMidHistoryBreakpoint` in `plugins/dist/qtap-plugin-anthropic/provider.ts` places a second `cache_control` breakpoint on a message ~15 indices back from the tail when `messages.length >= 20`. The index is rounded down to the nearest multiple of 15 so the breakpoint stays at the same content block for several turns rather than rewriting cache every turn. Without this, Anthropic's 20-block lookback from the tail breakpoint silently misses cache on long tool-use-heavy turns. Same call also fixes a parallel bug in `streamMessage` that still used `find(m => m.role === 'system')` post-commit-68a9eba6 (so block 2 was being silently dropped on streaming requests); now uses `filter` and emits all system blocks identically to `sendMessage`.
- **Eval harness scaffold.** `__tests__/eval/cache-stability/` asserts hash stability for systemBlock1 / systemBlock2 / tools across synthetic 30-turn fixtures, with an append-only-history transition counter. Phase 3 will plug recorded `BuiltContext` snapshots into the same harness. `__tests__/eval/character-voice/README.md` documents the deferred Phase 3 voice-regression gate.

#### Prompt restructure: two system messages, off-scene characters move to Host history

- **Two system messages instead of one.** `buildContext` (`lib/chat/context-manager.ts`) now emits the chat character's identity stack as the first system message and the (now fully static) Identity Reminder as a second system message. Together they produce a long stable prefix that providers can prefix-cache; the previous monolithic single-message prompt mixed cacheable and turn-variable content in the same string and defeated caching.
- **Identity Reminder is now fully static.** `buildIdentityReinforcement` (`lib/chat/context/system-prompt-builder.ts`) no longer accepts `userName` or `otherParticipantNames` and no longer interpolates a per-turn "do not write dialogue for [list]" of active participants. Phrasing is now the static "Do not write dialogue, actions, or thoughts for any other character." Other-participant attribution comes from the existing Host roster announcements + per-message `name` fields. Removed the unused `userName` / `otherCharacterNames` arguments at the help-chat call site (`lib/help-chat/system-prompt-builder.ts`). Test suite updated.

New helpers in `lib/services/host-notifications/writer.ts`: `buildOffSceneCharactersContent`, `findIntroducedOffSceneCharacterIds`, `postHostOffSceneCharactersAnnouncement`. #### Fix: scope:project diverged from the project's official document store

- **Root cause.** `convert-project-files-to-document-stores-v1` (April 23) moved every project's on-disk file directory into a database-backed mount point and `add-project-official-mount-point-v1` (April 25) wrote that mount's id into `projects.officialMountPointId`. Prospero already advertises this mount as the canonical "project-official" store and tells the LLM that `scope: 'project'` is an alias for it. But `lib/doc-edit/path-resolver.ts:resolveProjectPath` was never updated — it kept resolving `scope: 'project'` to `<filesDir>/<projectId>/<path>` on disk. Every `doc_write_file({ scope: 'project' })`, `doc_create_folder`, etc. silently recreated the legacy directory and wrote into it, while Document Mode UI / the Scriptorium read from the database mount. Files written by Friday looked fine to `doc_list_files` (which also walked the legacy fs) but were invisible to the UI.
- **Resolver fix.** `resolveProjectPath` now looks up `project.officialMountPointId`. When set, it dispatches through that mount: database-backed mounts return a `ResolvedPath` with `mountType: 'database'` and `mountPointId` populated (so `readFileWithMtime` / `writeFileWithMtimeCheck` route bytes through `lib/mount-index/database-store`); filesystem/Obsidian mounts return a path under `mountPoint.basePath`. Only when no official mount exists does it fall through to the legacy `<filesDir>/<projectId>/` layout.
- **Handler fixes.** Two doc-edit handlers bypass the path resolver and walk the legacy fs directly: `handleListFiles` and `handleGrep` in `lib/tools/handlers/doc-edit-handler.ts`. Both now check `resolveOfficialProjectMount(context.projectId)` first. When an official mount exists, `handleListFiles` enumerates from the database mount (only when `scope: 'project'` is explicitly requested — multi-scope mode picks it up via the document_store branch already), and `handleGrep` skips the legacy walk entirely. All other doc-edit handlers (`handleWriteFile`, `handleMoveFile`, `handleCopyFile`, `handleDeleteFile`, `handleCreateFolder`, `handleDeleteFolder`, `handleMoveFolder`) already dispatched on `resolved.mountType === 'database'`, so they pick up the new resolution automatically.
- **Reabsorb migration.** New migration `reabsorb-leftover-project-files-v1` (`migrations/scripts/reabsorb-leftover-project-files.ts`) handles instances where the divergence already happened. For each project with `officialMountPointId` set and a leftover directory at `<filesDir>/<projectId>/`, it walks the directory, imports any file not already in the database mount (compared by `relativePath`), and renames the directory to a fresh `_doc_store_archive_v2` (collision-suffixed up to `_v99`, then UUID). Skips projects whose official mount is filesystem-backed (path semantics already match) and projects with no official mount (resolver fallback is correct for them). Idempotent — re-running on an instance that's already been handled is a no-op because the directory is gone.
- **Concrete impact (Friday).** Project `Quilltap Plans` (`f29e2112-...`) had three Scenarios files written via `doc_write_file({ scope: 'project' })` on April 29 that landed in `~/iCloud/Quilltap/Friday/files/f29e2112-.../Scenarios/` but were absent from the `Project Files: Quilltap Plans` mount (`4c1a9cf6-...`). The Document Mode picker couldn't see them. Resolver fix alone makes new writes land in the right place; the reabsorb migration moves the existing leftover Scenarios + Feature Requests + Folio Drafts files into the mount.

#### Fix: backup + restore failed with "Invalid string length" on large instances

- **Root cause (write side).** `writeJsonFile` in `lib/backup/backup-service.ts` called `JSON.stringify(data, null, 2)`, which returns a single string. With ~17k LLM-log rows totalling ~720 MB of `request`/`response` text, the encoded string exceeded V8's ~512 MB max-string limit and threw `RangeError: Invalid string length`. The route returned a generic 500 and the client logged `Backup creation failed {}`.
- **Fix (write side).** Added `writeJsonArrayFile` that streams the array to disk via `Readable.from` + `pipeline`, encoding one element at a time so each `JSON.stringify` call stays bounded. All twenty data-array writes now use it; `writeJsonFile` is reserved for the small manifest. Output format is byte-identical to the previous pretty-printed JSON.
- **Root cause (read side).** `readJsonFile` in `lib/backup/restore-service.ts` did `fs.readFile(path, 'utf8')` followed by `JSON.parse(content)`. Both steps materialize the whole file as one string, so a large `llm-logs.json` would throw `ERR_STRING_TOO_LONG` before parsing even started — making backups created on large instances impossible to import.
- **Fix (read side).** Added `readJsonArrayFile`, a JSON-aware streaming scanner that reads the file in 1 MB chunks, tracks string/escape state and brace/bracket depth to find top-level element boundaries, and `JSON.parse`s one element at a time. Per-element memory stays bounded by individual row size (a few MB at most). All twenty data-array reads in `parseBackupZip` route through it (with a `readJsonArrayFileOptional` variant for back-compat optional files); `readJsonFile` is kept for the manifest. Smoke-checked against escaped quotes, `}`/`]` inside strings, nested objects, unicode, and multi-MB elements crossing chunk boundaries.

#### Inter-character memory fetch: SQLite ranks and caps each partition

- **Window function instead of unbounded fetch.** `MemoriesRepository.findByCharacterAboutCharacters` (`lib/database/repositories/memories.repository.ts`) now takes a required `limitPerCharacter` and runs a `ROW_NUMBER() OVER (PARTITION BY aboutCharacterId ORDER BY importance DESC, COALESCE(lastReinforcedAt, createdAt) DESC)` CTE through `rawQuery`. Each partition is ranked and capped server-side, so we decode a bounded number of rows per about-character instead of every matching memory. Hand-hydration handles the JSON-encoded array columns (`keywords`, `tags`, `relatedMemoryIds`) since the rawQuery path bypasses `SQLiteCollection.find`. The `embedding` BLOB rides through `MemorySchema.safeParse` directly.
- **Tie-break key.** Sort key approximates `calculateEffectiveWeight` so a recently-reinforced memory beats a stale one of equal raw importance — the formatter still re-ranks by effective weight in JS over the much smaller post-CTE set.
- **Budget-driven sizing.** `lib/chat/context-manager.ts` derives the per-character cap from the inter-character whisper budget: `maxEntries = ceil(interCharacterBudget / 40)`, `maxEntriesPerCharacter = max(20, ceil(maxEntries / otherChars) × 3)`. The 3× buffer absorbs (a) effective-weight re-ranking diverging from raw importance order and (b) shorter-than-average lines packing tighter than the 40-token estimate.
- **Concrete impact (Friday + Charlie + Amy).** Old: 10,081 rows decoded, importance down to 0.35. New (200k-context budget, 150 per character): 600 rows, all at importance ≥ 0.95 — the same rows that would have made it through the budget cut anyway.

#### Fix: self-referential memories were being labelled "about the user"

- **Root cause.** `lib/memory/memory-processor.ts` ran two parallel auto-extraction passes per turn — `extractMemoryFromMessage` (about the user) and `extractCharacterMemoryFromMessage` (self-referential, about the character themselves) — but both branches called the same `createMemoryFromCandidate` helper, which hard-coded `aboutCharacterId: ctx.userCharacterId || null`. Every self-extracted memory was therefore written as if it were about the user (or null when no user-character was resolved), polluting the inter-character signal in the data and skewing optimizer / recall ranking.
- **Helper split.** `createMemoryFromCandidate` is replaced by two helpers in `memory-processor.ts`: `createUserMemoryFromCandidate` (about-target = `ctx.userCharacterId`) and `createSelfMemoryFromCandidate` (about-target = `ctx.characterId`). Each extraction branch wires to its own helper.
- **Skip user-memory pass when no user-character is attached.** When `resolveUserIdentity` does not produce a `characterId`, the user-memory extraction call is bypassed entirely instead of fabricating a `null`-targeted memory. A debug log line and a one-time Host whisper (`systemKind: 'no-user-character'`) prompt the operator to attach or create a user persona. The whisper is idempotent — only one is posted per chat — and is suppressed in chats that already have one.
- **Name-presence safety net.** New module `lib/memory/about-character-resolution.ts` exports `nameAppears`, `namesForAboutCharacter`, and `resolveAboutCharacterId`. `createMemoryWithGate` (`lib/memory/memory-service.ts`) now applies the rule before insert: for AUTO memories with a non-self `aboutCharacterId`, if the about-character's name + aliases (plus generic `user` / `the user` for `controlledBy: 'user'` characters) do not appear in `summary + content`, `aboutCharacterId` is collapsed to the holder's id (self-reference). MANUAL memories pass through unchanged so explicit user attributions are honoured. Failures in the lookup never block a write.
- **Backfill migration.** `migrations/scripts/align-about-character-id.ts` (`align-about-character-id-v1`, depends on `sqlite-initial-schema-v1` and `rename-persona-columns-v1`) walks every memory and applies the same rule:
  - Memories where `aboutCharacterId` is set but the about-character's name set is absent from the text → flipped to the holder's id.
  - Memories where `aboutCharacterId IS NULL` → if exactly one user-controlled character's specific name/aliases appear, attributed to that persona; otherwise if generic `user`/`the user` appears AND there is exactly one user-controlled character system-wide, attributed to that persona; otherwise flipped to the holder.
  - Memories where the holder and the existing about-character are both named in the text are left as-is (the existing about-attribution wins).
  - Self-references (`aboutCharacterId === characterId`) and ambiguous null rows (multiple user-personas named, or generic-user with multiple personas) are left untouched.
  - Per-character flipped/attributed/untouched counts are logged at the migration's `info` level so post-run spot-checks don't need a separate query.
- **New host whisper helper.** `postHostNoUserCharacterAnnouncement` in `lib/services/host-notifications/writer.ts` posts the advisory whisper. It checks the chat's existing messages for a `systemSender: 'host'` row with `systemKind: 'no-user-character'` and is a no-op if one exists.
- **Tests.** New unit suite `__tests__/unit/lib/memory/about-character-resolution.test.ts` covers the name-presence rule, alias handling, and the user-controlled augmentation. The existing `memory-processor-regression.test.ts` was updated to (a) thread `userCharacterId` through `baseContext` so the user-memory pass still fires under the new gate, (b) mock the host-notifications writer module to avoid pulling in the database stack, and (c) add three new cases verifying the no-user-character skip, that self-pass writes `aboutCharacterId === characterId`, and that user-pass writes `aboutCharacterId === userCharacterId`.

### 4.3.1

#### Bug Fixes

- **SQLite journal mode default changed from WAL to TRUNCATE**: Quilltap data directories are commonly placed inside cloud-synced folders (iCloud Drive, Dropbox, OneDrive, Google Drive). WAL mode keeps `.db-wal` and `.db-shm` files alongside the main `.db`; if those files sync out of order with the main database — which all four providers can do, especially on dirty shutdown — the database can be corrupted or lose recent writes on the next open. Switching to `TRUNCATE` keeps the rollback journal in a single auxiliary file that is truncated to zero on every commit, eliminating the multi-file sync hazard. Existing databases auto-migrate on first open after upgrade (SQLite checkpoints any pre-existing WAL into the main file as part of the journal-mode transition). The change applies to all three databases — main (`quilltap.db`), LLM logs (`quilltap-llm-logs.db`), and mount index (`quilltap-mount-index.db`) — plus the meta-table connection used during startup.
- **`SQLITE_WAL_MODE` env var inverted**: Previously `SQLITE_WAL_MODE=false` was the opt-out; now `SQLITE_WAL_MODE=true` is the opt-in. Set this only when the data directory lives on a fast local SSD that is not synced to the cloud.
- **Internal**: `lib/startup/db-encryption-converter.ts` now leaves freshly-encrypted databases in TRUNCATE mode rather than WAL, matching the new resting state.
